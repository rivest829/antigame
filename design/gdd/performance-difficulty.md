# 演出难度系统 (Performance Difficulty System)

> **Status**: In Design
> **Author**: [user + agents]
> **Last Updated**: 2026-04-27
> **Implements Pillar**: 选择是有意义的, 成长是可见的

## Overview

演出难度系统是《地下排练室》的**演出挑战量化中枢**，负责将乐队的当前能力、选择的曲目、以及演出场地的特性转化为一个具体的"难度评分"。这个评分不是给玩家直接看的隐藏数值，而是演出判定系统用来计算演出成败、评分等级和声誉收益的核心输入。

**系统边界**：
- **包含**：难度计算公式、场地-曲目-能力匹配度评估、难度等级划分、动态难度调整（观众期待值）
- **不包含**：演出判定的具体骰子/随机机制（属于演出判定系统）、声誉收益的具体数值（属于声誉系统）、曲目静态数据（属于曲目数据库）、场地静态数据（属于内容数据库）

**玩家如何感知**：
- 选择高 tier 场地演出低熟练度曲目 → 演出艰难，可能失败
- 选择匹配乐队能力的场地和曲目 → 演出顺利，获得好评
- 同一首曲子，在校园舞台轻松完成，在 Livehouse 却失误频出 → "场地选择是策略的一部分"

**为什么必须存在**：
没有演出难度系统，所有演出都会变成"按按钮收奖励"——没有挑战，没有风险，也就没有成就感。它是"选择是有意义的"和"成长是可见的"两大支柱的交汇点：你不仅要练得好，还要选得对。

## Player Fantasy

> **"演出不是考试——它是赌注。你押上的是这支乐队的名声，赌的是台下那些人会不会记住你们。"**

你站在校园开放舞台的后台，手里攥着 setlist。三首歌：《地下室之光》《锈》《404》。你知道《地下室之光》闭着眼睛都能弹——你们练了太多次。但《404》呢？上周才解锁，熟练度才 32%。大伟说他没问题，林野没说话（林野从来不在演出前说话）。

你有两个选择：
1. **安全路线**：只演《地下室之光》和《锈》，稳妥，不会出错，但也不会让人记住
2. **冒险路线**：加上《404》，风险高——如果成功了，这可能是乐队第一个"有记忆点"的演出；如果失败，你在校园里的口碑会受损

你想起内容数据库里的场地信息：校园开放舞台的 `difficulty_modifier` 是 1.0，观众容量 50 人，声誉奖励基础值低。这里不是一个能一夜成名的地方，但这里是你们的第一步。

你最终选了安全路线。演出顺利进行，观众鼓掌，但没有安可。你看着声誉条涨了一小截，心想：**下一次，等《404》熟练度到 60%，我们要试试本地 Livehouse。**

那一刻你感受到的不是"数值最优解"——而是**"我在为这支乐队做真正的决定，这些决定有后果"**。

**核心情感时刻**：
当你第一次带着 80% 熟练度的《锈》走进本地 Livehouse，演出判定给出"出色"评级时，你回想起第一次在地下室磕磕绊绊弹这首歌的场景。从 15% 到 80%，从地下室到 Livehouse——**这个难度系统不是挡在你面前的墙，它是你爬过的山。**

**参考体验**：
- 《足球经理》的战术风险——选择保守或激进都有后果，但后果是叙事性的，不只是数值性的
- 《太吾绘卷》的功法突破——成功率不是隐藏的，你清楚地知道风险，然后决定要不要赌

**这不是"我必须赢"的幻想——这是"我选择怎么赢"的幻想。**

## Detailed Design

### Core Rules

#### 1. 难度评分 (Difficulty Score)

演出难度用一个 0-100 的浮点数表示，供演出判定系统使用：

```
difficulty_score = base_difficulty × venue_modifier × readiness_penalty × morale_factor
```

**难度等级划分**：

| 难度评分范围 | 等级 | 演出判定影响 | 玩家感知 |
|-------------|------|------------|---------|
| 0-30 | 轻松 | 几乎不可能失败 | "这场我们稳了" |
| 31-50 | 适中 | 正常挑战 | "有点紧张，但没问题" |
| 51-70 | 困难 | 高风险高回报 | "这可能是我们的突破" |
| 71-90 | 极限 | 需要运气 + 完美状态 | "要么传奇，要么灾难" |
| 91-100 | 不可能 | 几乎必然失败 | "我们还没准备好" |

#### 2. 基础难度 (Base Difficulty)

基础难度由曲目本身决定：

```
base_difficulty = song_difficulty_rating × proficiency_requirement
```

**曲目难度等级** (`song_difficulty_rating`)：

| 难度 | 值 | 说明 |
|------|-----|------|
| 简单 | 15 | 基础曲目，新手友好 |
| 普通 | 30 | 标准曲目，需要一定练习 |
| 困难 | 50 | 复杂编排，需要熟练配合 |
| 专家 | 75 | 技术挑战，容错极低 |
| 传奇 | 95 | 巅峰之作，极少乐队能驾驭 |

**熟练度要求** (`proficiency_requirement`)：

```
proficiency_requirement = 1.0 - (current_proficiency / 100) × 0.5
```

- 熟练度 0% → requirement = 1.0（满额难度）
- 熟练度 100% → requirement = 0.5（难度减半）

**为什么熟练度降低难度**：曲子练得越熟，演出时越不容易出错，即使技术难度不变，实际执行难度下降了。

#### 3. 场地修正 (Venue Modifier)

来自内容数据库的场地参数：

```
venue_modifier = difficulty_modifier × (1 + (venue_tier - 1) × 0.1)
```

| 场地 Tier | 基础修正 | Tier 加成 | 总修正范围 |
|-----------|---------|----------|-----------|
| 1 (地下室) | 0.8 | 0 | 0.8 |
| 2 (校园) | 1.0 | +0.1 | 1.1 |
| 3 (Livehouse) | 1.2 | +0.2 | 1.4 |
| 4 (音乐节) | 1.4 | +0.3 | 1.7 |
| 5 (万人馆) | 1.6 | +0.4 | 2.0 |

**场地修正上限**：`venue_modifier` 最高不超过 2.0。

#### 4. 准备度惩罚 (Readiness Penalty)

评估乐队是否"准备好"演出这首曲目：

```
readiness_penalty = max(1.0, sum(instrument_gaps) / instrument_count)
```

**乐器差距计算**：

对于曲目所需的每种乐器：
```
instrument_gap = max(0, required_skill - actual_skill)
```

- `required_skill`：曲目对该乐器的要求（来自曲目数据库）
- `actual_skill`：参与演出角色在该乐器的技能值

**未覆盖乐器惩罚**：
如果乐队没有角色覆盖某所需乐器：
```
uncovered_penalty = 1.5  # 50% 难度激增
```

#### 5. 士气因子 (Morale Factor)

角色状态影响演出发挥：

```
morale_factor = 1.0 + ((50 - average_morale) / 100)
```

- 士气 100 → factor = 0.5（难度减半，超常发挥）
- 士气 50 → factor = 1.0（正常）
- 士气 0 → factor = 1.5（难度增加 50%，失常发挥）

**士气计算**：参与演出的所有角色的士气平均值。

#### 6. 观众期待值 (Audience Expectation)

动态调整：声誉越高，观众期待越高，相同客观难度下"感知难度"增加。

```
audience_expectation = 1.0 + (reputation / max_reputation) × 0.3
```

- 声誉 0 → expectation = 1.0
- 声誉满值 → expectation = 1.3（+30% 感知难度）

**最终难度评分**：
```
final_difficulty = min(100, difficulty_score × audience_expectation)
```

#### 7. 难度预览机制

玩家在确认演出前，系统提供**难度预览**：

| 预览等级 | 条件 | UI 显示 |
|---------|------|---------|
| 🟢 轻松 | final_difficulty ≤ 30 | 绿色标签 "十拿九稳" |
| 🟡 适中 | 31 ≤ final_difficulty ≤ 50 | 黄色标签 "正常发挥" |
| 🟠 困难 | 51 ≤ final_difficulty ≤ 70 | 橙色标签 "有点挑战" |
| 🔴 极限 | 71 ≤ final_difficulty ≤ 90 | 红色标签 "背水一战" |
| ⚫ 疯狂 | final_difficulty > 90 | 黑色标签 "自寻死路" |

**注意**：预览基于当前乐队状态。如果玩家设置演出后、实际演出前角色状态恶化（如士气下降），实际难度可能高于预览。

#### 8. 演出长度影响

演出的曲目数量影响难度：

```
set_length_factor = 1.0 + (song_count - 1) × 0.15
```

- 1 首歌：1.0×
- 2 首歌：1.15×
- 3 首歌：1.30×
- 4 首歌：1.45×

**体力衰减**：每多一首歌，角色精力消耗增加，可能间接影响士气因子。

### States and Transitions

演出难度系统本身是无状态的计算服务，但在演出流程中有以下阶段：

| 阶段 | 触发条件 | 系统行为 |
|------|---------|---------|
| **预览** | 玩家选择曲目和场地 | 计算并显示难度预览 |
| **锁定** | 玩家确认演出安排 | 记录当时的难度值作为"基准难度" |
| **执行** | 演出开始时 | 基于当前实际状态重新计算"实际难度" |
| **结算** | 演出结束时 | 将实际难度传给演出判定系统 |

**关键设计**：预览难度 ≠ 实际难度。玩家确认后，角色状态仍可能变化（如离线期间的事件影响士气），最终演出以实际状态为准。这让"提前安排演出"有风险——你不能在角色情绪低落时安排一周后的重要演出而不做点什么。

## Formulas

### 1. 基础难度公式 (Base Difficulty)

```
base_difficulty = song_difficulty_rating × proficiency_requirement
```

**Variables:**
| Variable | Symbol | Type | Range | Description |
|----------|--------|------|-------|-------------|
| song_difficulty_rating | sdr | int | 15-95 | 曲目基础难度等级 |
| current_proficiency | prof | float | 0-100 | 乐队对该曲目的熟练度百分比 |
| proficiency_requirement | pr | float | 0.5-1.0 | 熟练度对难度的减免系数 |

**Proficiency Requirement Sub-formula:**
```
proficiency_requirement = 1.0 - (current_proficiency / 100) × 0.5
```

**Output Range:** 7.5 到 95
**Example:** 曲目难度 50（困难），熟练度 60% → `base = 50 × (1.0 - 0.6 × 0.5) = 50 × 0.7 = 35`

---

### 2. 场地修正公式 (Venue Modifier)

```
venue_modifier = min(2.0, difficulty_modifier × (1 + (venue_tier - 1) × 0.1))
```

**Variables:**
| Variable | Symbol | Type | Range | Description |
|----------|--------|------|-------|-------------|
| difficulty_modifier | dm | float | 0.8-1.6 | 场地基础难度修正（来自内容数据库） |
| venue_tier | vt | int | 1-5 | 场地层级 |

**Output Range:** 0.8 到 2.0
**Example:** Livehouse（tier 3, dm 1.2）→ `modifier = 1.2 × (1 + 0.2) = 1.44`

---

### 3. 准备度惩罚公式 (Readiness Penalty)

```
readiness_penalty = max(1.0, average_instrument_gap / 20)
```

**Instrument Gap Sub-formula（每种乐器）:**
```
instrument_gap = max(0, required_skill - actual_skill)
```

**Variables:**
| Variable | Symbol | Type | Range | Description |
|----------|--------|------|-------|-------------|
| required_skill | req | int | 1-100 | 曲目对该乐器的要求 |
| actual_skill | act | int | 0-100 | 演出角色在该乐器的实际技能 |
| average_instrument_gap | gap | float | 0-100 | 所有所需乐器差距的平均值 |

**未覆盖乐器惩罚:**
如果某乐器无人演奏：`readiness_penalty = readiness_penalty × 1.5`

**Output Range:** 1.0 到 5.0+（未覆盖乐器时更高）
**Example:** 曲目需要吉他(要求40)、鼓(要求30)。林野吉他42，大伟鼓25。`gap = ((0 + 5) / 2) = 2.5` → `penalty = max(1.0, 2.5/20) = 1.0`（无惩罚）
**Example 2:** 吉他要求60，林野吉他45。`gap = 15` → `penalty = max(1.0, 15/20) = 1.0`... 等等，这个范围太小了。

让我重新调整。差距应该更敏感：

```
readiness_penalty = max(1.0, 1.0 + average_instrument_gap / 10)
```

**Revised Example:** 吉他要求60，林野吉他45。`gap = 15` → `penalty = max(1.0, 1.0 + 1.5) = 2.5`（难度变为2.5倍）

---

### 4. 士气因子公式 (Morale Factor)

```
morale_factor = 1.0 + ((50 - average_morale) / 100)
```

**Variables:**
| Variable | Symbol | Type | Range | Description |
|----------|--------|------|-------|-------------|
| average_morale | am | float | 0-100 | 参与演出角色的平均士气 |

**Output Range:** 0.5 到 1.5
**Example:** 平均士气 70 → `factor = 1.0 + ((50 - 70) / 100) = 1.0 - 0.2 = 0.8`（难度降低20%）
**Example 2:** 平均士气 30 → `factor = 1.0 + ((50 - 30) / 100) = 1.2`（难度增加20%）

---

### 5. 观众期待值公式 (Audience Expectation)

```
audience_expectation = 1.0 + (reputation / max_reputation) × 0.3
```

**Variables:**
| Variable | Symbol | Type | Range | Description |
|----------|--------|------|-------|-------------|
| reputation | rep | int | 0-max | 当前声誉值 |
| max_reputation | max | int | ≥100 | 声誉系统定义的上限 |

**Output Range:** 1.0 到 1.3
**Example:** 声誉 50，上限 200 → `expectation = 1.0 + (50/200) × 0.3 = 1.075`

---

### 6. 最终难度评分公式 (Final Difficulty Score)

```
difficulty_score = base_difficulty × venue_modifier × readiness_penalty × morale_factor
final_difficulty = min(100, difficulty_score × audience_expectation)
```

**完整计算示例:**

**场景**: 乐队要演出《锈》（难度 30），在本地 Livehouse（tier 3, dm 1.2）。乐队熟练度 55%。林野吉他 48（要求 40），大伟鼓 28（要求 25）。平均士气 65。

1. `base = 30 × (1.0 - 0.55 × 0.5) = 30 × 0.725 = 21.75`
2. `venue = 1.2 × (1 + 0.2) = 1.44`
3. `gap_guitar = max(0, 40 - 48) = 0`; `gap_drums = max(0, 25 - 28) = 0`; `avg_gap = 0` → `penalty = 1.0`
4. `morale = 1.0 + ((50 - 65)/100) = 0.85`
5. `difficulty = 21.75 × 1.44 × 1.0 × 0.85 = 26.6`
6. 假设声誉 30，上限 200：`expectation = 1.0 + 0.15 × 0.3 = 1.045`
7. `final = min(100, 26.6 × 1.045) = 27.8` → **🟢 轻松**

**困难场景示例:**
同一首《锈》在万人场馆（tier 5, dm 1.6）。熟练度 35%。林野吉他 35（要求 40），大伟鼓 20（要求 25）。平均士气 40。

1. `base = 30 × (1.0 - 0.35 × 0.5) = 30 × 0.825 = 24.75`
2. `venue = 1.6 × (1 + 0.4) = 2.24 → capped to 2.0`
3. `gap_guitar = 5`; `gap_drums = 5`; `avg = 5` → `penalty = 1.0 + 5/10 = 1.5`
4. `morale = 1.0 + ((50 - 40)/100) = 1.1`
5. `difficulty = 24.75 × 2.0 × 1.5 × 1.1 = 81.7`
6. 声誉 80，上限 200：`expectation = 1.0 + 0.4 × 0.3 = 1.12`
7. `final = min(100, 81.7 × 1.12) = 91.5` → **⚫ 疯狂**

## Edge Cases

- **If 曲目熟练度为 0%**: `proficiency_requirement = 1.0`，基础难度不减免。这是正确的——完全不会弹的曲子演出难度就是满额。
- **If 乐队能力远超曲目要求**: `readiness_penalty = 1.0`（不降低到 1.0 以下），防止出现"负难度"。即使乐队再强，演出也有基础挑战。
- **If 士气 > 100（存在 Buff）**: `morale_factor` 可以低于 0.5，最高士气 150 时 factor = 0.0。但难度评分不会低于 `base_difficulty × 0.1`，保留 10% 基础挑战。
- **If 未覆盖乐器且其他乐器也低于要求**: 惩罚叠加——先计算 `readiness_penalty`，再乘以 1.5。例如 gap=10 时 penalty=2.0，未覆盖后变为 3.0。
- **If 所有角色精力 ≤ 20**: 系统不允许安排演出（放置排练系统的规则），所以此情况不会在演出难度计算中出现。
- **If 声誉系统尚未实现（MVP 早期）**: `audience_expectation` 固定为 1.0，等声誉系统上线后接入。
- **If 场地 tier 或 difficulty_modifier 数据缺失**: 使用默认值 `tier=1`, `dm=1.0`，记录警告日志。内容数据库必须保证数据完整性。
- **If 演出前角色状态剧变（如离线期间触发负面事件导致士气从 70 跌到 20）**: 以实际演出时的状态重新计算。玩家会看到预览难度和实际难度不同——这是设计意图，提醒玩家"状态管理也是策略"。
- **If 同一角色同时演奏多个乐器**: 取该角色最高技能值作为 `actual_skill`。例如林野同时弹吉他和贝斯，曲目需要吉他——用吉他技能；需要贝斯——用贝斯技能。

## Dependencies

### 输入依赖（本系统需要的数据）

| 来源系统 | 输入数据 | 作用 |
|----------|----------|------|
| 曲目数据库 | 曲目难度等级、所需乐器、技能要求 | 计算基础难度和准备度 |
| 内容数据库 | 场地 difficulty_modifier、venue_tier | 计算场地修正 |
| 角色属性系统 | 角色乐器技能值、士气值 | 计算准备度和士气因子 |
| 放置排练系统 | 曲目熟练度百分比 | 计算熟练度减免 |
| 声誉系统 | 当前声誉值、声誉上限 | 计算观众期待值 |

### 输出依赖（使用本系统数据的系统）

| 目标系统 | 输出数据 | 作用 |
|----------|----------|------|
| 演出判定系统 | final_difficulty（0-100） | 演出成败、评分等级的核心输入 |
| 声誉系统 | 难度等级（轻松/适中/困难/极限/疯狂） | 影响声誉收益倍率 |
| 进度解锁系统 | 演出结果（成功/失败） | 某些解锁条件依赖演出成功 |
| 叙事事件系统 | 演出结果 | 触发特定演出相关事件 |

### 双向依赖说明

- **演出判定系统**：演出难度系统的唯一核心消费者。演出判定系统定义了" difficulty_score 如何转化为演出结果"，但具体的 difficulty_score 计算完全由本系统负责。
- **声誉系统**：双向弱依赖。声誉影响观众期待值，同时难度等级影响声誉收益。循环依赖通过"快照"避免：演出前锁定声誉值，演出后统一结算。

### 依赖顺序

```
角色属性系统 ──┐
曲目数据库 ────┼──→ 演出难度系统 ──→ 演出判定系统
内容数据库 ────┤         ↓
放置排练系统 ──┘    声誉系统（循环，通过快照解耦）
```

## Tuning Knobs

| 参数 | 当前值 | 安全范围 | 调整影响 | 所在公式 |
|------|--------|---------|---------|---------|
| `proficiency_discount_rate` | 0.5 | 0.3-0.7 | 熟练度对难度的减免程度。越高=熟练越重要 | `proficiency_requirement` |
| `tier_bonus_per_level` | 0.1 | 0.05-0.15 | 场地 tier 对难度的加成。越高=场地越重要 | `venue_modifier` |
| `venue_modifier_cap` | 2.0 | 1.5-3.0 | 场地修正上限。越低=顶级场地越"可及" | `venue_modifier` |
| `readiness_gap_divisor` | 10 | 5-20 | 技能差距转化为惩罚的敏感度。越小=差距惩罚越重 | `readiness_penalty` |
| `uncovered_penalty_multiplier` | 1.5 | 1.2-2.0 | 未覆盖乐器的惩罚强度。越高=阵容完整性越重要 | `readiness_penalty` |
| `morale_sensitivity` | 0.01 | 0.005-0.02 | 士气每点变化对难度的影响。越高=士气越关键 | `morale_factor` |
| `audience_expectation_max` | 0.3 | 0.1-0.5 | 声誉满值时的额外难度百分比。越高=高声誉越难维持 | `audience_expectation` |
| `set_length_increment` | 0.15 | 0.1-0.25 | 每增加一首歌的难度增幅。越高=长 setlist 越冒险 | `set_length_factor` |
| `minimum_difficulty_ratio` | 0.1 | 0.05-0.2 | 难度最低不会低于基础难度的比例。越高=再强也有挑战 | `final_difficulty` |

**调优策略**：
- 如果玩家普遍选择保守路线（只打轻松演出）：降低 `readiness_gap_divisor` 或提高 `venue_modifier_cap`，让"跳级"更有吸引力
- 如果玩家频繁失败感到挫败：提高 `proficiency_discount_rate` 或降低 `audience_expectation_max`
- 如果长 setlist 无人问津：降低 `set_length_increment`

## Visual/Audio Requirements

### 视觉需求

**难度预览标签**：
- 🟢 轻松：绿色，柔和圆角标签，图标为微笑表情
- 🟡 适中：黄色，标准标签，图标为平静表情
- 🟠 困难：橙色，略尖锐标签，图标为紧张表情
- 🔴 极限：红色，锯齿边缘标签，图标为流汗表情
- ⚫ 疯狂：深灰/黑色，破碎边缘标签，图标为骷髅/警告

标签风格应符合垃圾摇滚美学——不是干净的游戏UI元素，而是像手写在胶带或破纸上的标签。

**难度分解可视化（可选 MVP+）**：
玩家 hover/点击难度标签时，展开一个迷你图表显示：
- 曲目难度贡献（蓝色条）
- 场地修正贡献（红色条）
- 准备度影响（绿色/黄色条）
- 士气影响（紫色条）

**演出前状态面板**：
显示"预览难度 vs 实际难度"对比（如果两者不同），用箭头指示变化方向。

### 音频需求

**难度标签音效**：
- 轻松：轻快的拨弦声
- 适中：标准鼓点
- 困难：略带紧张的吉他刮弦
- 极限：沉重的底鼓
- 疯狂：失真噪音

**状态变化音效**：
如果实际难度高于预览难度（演出前状态恶化），播放一个不和谐的音效提示玩家"情况有变"。

## UI Requirements

### 演出安排界面

**难度预览区域**：
- 位置：曲目和场地选择下方，确认按钮上方
- 内容：
  - 难度标签（🟢🟡🟠🔴⚫）+ 文字描述
  - 简短提示语（如 "这支乐队驾驭这场演出十拿九稳" 或 "这可能是一场灾难"）
  - 风险/收益摘要（如 "失败风险：低 / 声誉收益：中"）

**演出确认对话框**：
- 显示最终确认信息：场地 + 曲目 + 难度标签
- 如果难度为 🔴 极限 或 ⚫ 疯狂：增加确认提示"你确定要冒这个险吗？"
- 确认后进入"已锁定"状态，显示倒计时/日期到演出时间

**演出前提醒（如果实际难度变化）**：
- 如果演出前角色状态恶化导致难度上升：弹出提醒
- 显示：预览难度标签 → 实际难度标签，箭头指示变化
- 提供选项："继续演出" / "推迟演出"（返回排练）

### 难度预览的交互设计

| 交互 | 行为 |
|------|------|
| 点击难度标签 | 展开/收起难度分解详情 |
| 更换曲目 | 实时更新难度预览 |
| 更换场地 | 实时更新难度预览 |
| 添加/移除曲目 | 实时更新难度预览（考虑 set_length_factor） |

## Acceptance Criteria

### 功能测试

- [ ] 选择曲目 + 场地后，难度预览在 100ms 内显示
- [ ] 更换曲目或场地，难度预览实时更新
- [ ] 相同曲目在 tier 1 场地和 tier 5 场地，难度差异 ≥ 50%
- [ ] 熟练度 0% vs 100% 的同一曲目，基础难度差异 = 50%
- [ ] 未覆盖乐器时，难度显著上升（≥ 1.5×）
- [ ] 士气 0 vs 100，难度差异 = 1.5× vs 0.5×
- [ ] 难度评分始终限制在 0-100 范围内
- [ ] 难度标签正确映射到 5 个等级区间

### 边界测试

- [ ] 乐队能力远超要求 → 难度不低于基础难度的 10%
- [ ] 所有乐器未覆盖 → 惩罚正确叠加
- [ ] 声誉系统未实现时 → audience_expectation = 1.0
- [ ] 场地数据缺失 → 使用默认值，记录日志

### 集成测试

- [ ] 从曲目数据库读取难度参数 → 计算正确
- [ ] 从内容数据库读取场地参数 → 计算正确
- [ ] 从角色属性系统读取技能和士气 → 计算正确
- [ ] 输出 final_difficulty 到演出判定系统 → 值在预期范围
- [ ] 演出前状态变化 → 实际难度正确更新

### 性能测试

- [ ] 难度计算耗时 < 1ms（纯数学运算，无 I/O）
- [ ] 连续 100 次计算 → 结果一致（确定性）

## Open Questions

- **演出长度 factor 的精力消耗如何量化？** 目前只在难度公式中体现，但角色精力实际下降多少由哪个系统计算？
- **配合技（Combo）是否影响难度？** 例如触发"倔强节奏"后，是否降低鼓+吉他的技能要求？
- **化学反应系统上线后，是否增加"默契因子"？** 高默契角色组合演出时难度降低？
- **是否需要"演出历史"影响难度？** 同一首曲子反复演出同一场地，是否因"熟悉场地"而降低难度？
- **多人乐队（3-4人）时，如何计算 average_morale？** 是所有参与角色平均，还是加权（主唱/核心角色权重更高）？
