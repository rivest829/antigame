# 内容数据库 (Content Database)

> **Status**: Designed (pending review)
> **Author**: [user + agents]
> **Last Updated**: 2026-04-23
> **Implements Pillar**: 成长是可见的

## Overview

内容数据库是《地下排练室》的**可解锁内容仓库**，定义了所有非角色、非曲目的游戏内容及其参数。它是进度解锁系统的数据基础（告诉系统"有哪些东西可以解锁"），也是演出难度系统和声誉系统的参数来源（提供场地难度、声誉奖励等数值）。

**系统边界**：
- **包含**：演出场地定义、配合技定义、剧情章节定义、各类内容的解锁条件与参数
- **不包含**：解锁条件的实时判定逻辑（属于进度解锁系统）、内容的视觉呈现（属于美术资产）、配合技的动态触发计算（属于化学反应系统）、叙事文本内容（属于叙事事件系统）

**玩家如何感知**：
- 看到"解锁了新场地：校园开放舞台" → "乐队终于有地方演出了"
- 发现林野和大伟一起排练时触发"倔强节奏" → "他们俩配合起来有奇效"
- 剧情章节推进 → "故事在继续，这支乐队在经历什么"

**为什么必须存在**：
没有内容数据库，进度解锁系统就不知道"解锁什么"，演出难度系统就不知道"在哪里演出有多难"，声誉系统就不知道"达到什么等级给什么奖励"。它将抽象的"进度"转化为具体、可命名的内容目标，是实现"成长是可见的"支柱的核心数据层。

## Player Fantasy

玩家应该感受到：**"这个世界在向我们敞开——每一次解锁都是一个新的舞台、一种新的可能。"**

你第一次点击"解锁校园开放舞台"时，看到的不是一行代码或一个数值变化。你看到的是乐队从漏水的地下室走到了一个有观众的房间。你看到的是希望。

**核心情感时刻**：
当你解锁"本地 Livehouse"时，你回想起乐队成立第一天在地下室练《地下室之光》的场景。你对比这两个画面——从无人知晓到有人买票来看你——那一刻你感受到的不是"数值涨了"，而是**"我们真的在成长"**。

**参考体验**：
- 《星露谷物语》的社区中心献祭——每次完成一个房间，小镇就多一点变化（我们的内容解锁也应让玩家感受到世界的回应）
- 《偶像梦幻祭》的场地解锁——从小剧场到巨蛋，每一个新场地都是身份的转变

**这不是"收集成就"的幻想**——这是"世界因你而扩大"的幻想。

## Detailed Design

### Core Rules

**1. 内容分类体系**

内容数据库采用**类型-条目**两级结构，支持未来扩展：

| 内容类型 | 类型标识 | MVP 数量 | 说明 |
|----------|---------|---------|------|
| 演出场地 (Venue) | `venue` | 1 个 | 乐队演出的地点，定义演出难度和收益 |
| 配合技 (Combo) | `combo` | 1 个 | 特定角色组合触发的特殊效果 |
| 剧情章节 (Story Chapter) | `chapter` | 2 个 | 叙事内容的章节节点 |

**2. 通用内容数据结构**

所有内容条目共享以下基础字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `content_id` | 字符串 | 唯一标识符，如 `venue_campus_stage` |
| `content_type` | 枚举 | `venue` / `combo` / `chapter` |
| `name` | 字符串 | 显示名称 |
| `description` | 文本 | 简短描述 |
| `unlock_conditions` | 条件数组 | 解锁条件（与曲目数据库相同的条件系统） |
| `is_hidden` | 布尔 | `true` 时玩家在解锁前看不到该内容的存在 |
| `unlock_order` | 整数 | 解锁顺序提示（用于 UI 排序和进度感） |

**3. 演出场地 (Venue) 专用字段**

| 字段 | 类型 | 说明 |
|------|------|------|
| `difficulty_modifier` | 浮点数 | 场地难度加成（乘数，1.0 为基准） |
| `audience_capacity` | 整数 | 观众容量，影响声誉收益上限 |
| `reputation_reward_base` | 整数 | 基础声誉奖励 |
| `reward_multiplier` | 浮点数 | 整体收益倍率 |
| `venue_tier` | 整数 | 场地层级（1=最小，5=最大） |

**场地层级与视觉效果映射**：
| 层级 | 场地规模 | 视觉特征 |
|------|---------|---------|
| 1 | 地下室/角落 | 昏暗、拥挤、只有几盏灯 |
| 2 | 校园/社区 | 小型舞台、简陋灯光、几十人观众 |
| 3 | 本地 Livehouse | 专业舞台、基础音响、几百人 |
| 4 | 音乐节 | 大型舞台、专业灯光音响、数千人 |
| 5 | 万人场馆 | 巨型舞台、炫目灯光、海量观众 |

**4. 配合技 (Combo) 专用字段**

| 字段 | 类型 | 说明 |
|------|------|------|
| `required_members` | 字符串数组 | 触发配合技需要的角色 ID 列表 |
| `trigger_condition` | 枚举 | 触发条件：`REHEARSAL`（排练时）/ `PERFORMANCE`（演出时）/ `ALWAYS`（ always active） |
| `effect_type` | 枚举 | 效果类型：`SKILL_BOOST`（技能成长加成）/ `CHEMISTRY_BOOST`（化学反应加成）/ `MOOD_BUFF`（心情增益） |
| `effect_value` | 浮点数 | 效果数值（如 +0.15 = +15%） |
| `effect_description` | 文本 | 玩家可见的效果描述 |

**5. 剧情章节 (Chapter) 专用字段**

| 字段 | 类型 | 说明 |
|------|------|------|
| `chapter_number` | 整数 | 章节序号（用于排序） |
| `chapter_arc` | 字符串 | 所属故事线（如 `main` 主线 / `character_林野` 个人线） |
| `linked_events` | 字符串数组 | 关联的叙事事件 ID 列表（由叙事事件系统定义） |
| `completion_reward` | 字典 | 章节完成奖励（如 `{声誉: 50, 心情: +10}`） |

**6. 解锁条件系统**

采用与曲目数据库相同的条件系统：

| 条件类型 | 参数 | 说明 |
|----------|------|------|
| `SKILL_THRESHOLD` | `instrument`, `min_skill` | 某乐器技能 ≥ X |
| `SONG_PROFICIENCY` | `song_id`, `min_proficiency` | 某曲目熟练度 ≥ Y% |
| `REPUTATION_MILESTONE` | `min_reputation` | 声誉 ≥ Z |
| `CONTENT_UNLOCKED` | `content_id` | 某内容已解锁 |
| `PERFORMANCE_COMPLETED` | `venue_id`, `min_score` | 在某场地演出且分数 ≥ W |

组合逻辑：默认 AND，支持 OR 和 NOT。

**7. MVP 内容清单**

**演出场地：**

| 内容 ID | 名称 | 层级 | 难度加成 | 观众容量 | 声誉基础奖励 | 解锁条件 | 说明 |
|---------|------|------|----------|----------|-------------|----------|------|
| `venue_campus_stage` | 校园开放舞台 | 2 | 1.0x | 50 | 10 | 完成首次排练（教程引导自动解锁） | MVP 唯一演出场地，乐队第一次面对观众 |

**配合技：**

| 内容 ID | 名称 | 所需角色 | 触发条件 | 效果类型 | 效果值 | 解锁条件 | 说明 |
|---------|------|----------|----------|----------|--------|----------|------|
| `combo_stubborn_rhythm` | 倔强节奏 | 林野, 大伟 | 排练时 | 技能成长加成 | +0.15 | 林野 + 大伟一起排练累计 5 小时 | 倔强的吉他 + 随和的鼓 = 意想不到的默契 |

**剧情章节：**

| 内容 ID | 名称 | 序号 | 故事线 | 关联事件 | 完成奖励 | 解锁条件 | 说明 |
|---------|------|------|--------|----------|----------|----------|------|
| `chapter_basement_light` | 地下室的光 | 1 | main | 开场事件 | 声誉 +10 | 自动解锁（游戏开始） | 乐队成立的第一天 |
| `chapter_first_show` | 第一次演出 | 2 | main | 首次演出事件 | 声誉 +20, 心情 +5 | `venue_campus_stage` 已解锁 | 从地下室走向舞台 |

### States and Transitions

每块内容在其生命周期中存在三种状态：

| 状态 | 触发条件 | 玩家可见效果 |
|------|----------|-------------|
| **锁定 (Locked)** | 解锁条件未满足 | 隐藏内容不显示；非隐藏内容显示为灰色，带锁定图标和解锁提示 |
| **已解锁 (Unlocked)** | 解锁条件满足 | 内容激活，玩家可使用/访问。场地可演出，配合技可触发，章节可推进 |
| **已完成 (Completed)** | 内容被完全使用/体验 | 仅适用于剧情章节（看完所有关联事件）和部分配合技（达到最大效果）。显示完成标记 |

**状态转换图**：

```
        解锁条件满足                    所有关联事件完成
   ┌──────────────────┐          ┌──────────────────┐
   │                  │          │                  │
   │     锁定         │─────────►│    已解锁        │─────────► 已完成
   │                  │          │                  │
   └──────────────────┘          └──────────────────┘
                                         ▲
                                         │
                                   解锁条件重新满足
                                   （已完成状态不可逆）
```

**不可逆规则**：内容解锁后不会重新锁定。剧情章节完成后不会回退。

**输入（本系统接收的数据）**：

| 来源系统 | 输入数据 | 作用 |
|----------|----------|------|
| 角色属性系统 | 乐器技能值 | 解锁条件判定 |
| 曲目数据库 | 曲目熟练度 | 解锁条件判定 |

| 声誉系统 | 当前声誉值 | `REPUTATION_MILESTONE` 条件判定 |

**输出（本系统提供的数据）**：

| 目标系统 | 输出数据 | 作用 |
|----------|----------|------|
| 演出难度系统 | 场地难度加成、观众容量 | 计算演出难度基准 |
| 声誉系统 | 场地声誉奖励基础值 | 计算演出后声誉收益 |
| 进度解锁系统 | 内容列表、解锁条件定义 | 判定哪些内容可以解锁 |
| 化学反应系统 | 配合技定义（角色组合、效果） | 判定何时触发配合技 |
| 叙事事件系统 | 章节定义、关联事件列表 | 驱动叙事进度 |
| UI 系统 | 内容列表、解锁状态、完成状态 | 界面展示 |

## Formulas

### 1. 场地声誉收益公式 (Venue Reputation Reward)

The venue reputation reward formula is defined as:

`reputation_reward = reputation_reward_base × performance_score_multiplier × venue_reward_multiplier`

**Variables:**
| Variable | Symbol | Type | Range | Description |
|----------|--------|------|-------|-------------|
| reputation_reward_base | base | int | 5–100 | 场地定义的基础声誉奖励 |
| performance_score_multiplier | score_mult | float | 0.5–2.0 | 演出评分倍率（来自演出判定系统：失败=0.5，及格=1.0，优秀=1.5，完美=2.0） |
| venue_reward_multiplier | venue_mult | float | 0.5–2.0 | 场地定义的收益倍率 |

**Output Range:** 2.5 到 400（声誉值）
**Example:** 校园开放舞台 base=10, 演出评分及格（score_mult=1.0）, venue_mult=1.0 → `reputation_reward = 10 × 1.0 × 1.0 = 10`

### 2. 场地难度基准公式 (Venue Difficulty Base)

The venue difficulty base formula is defined as:

`venue_difficulty = song_difficulty × venue_difficulty_modifier`

**Variables:**
| Variable | Symbol | Type | Range | Description |
|----------|--------|------|-------|-------------|
| song_difficulty | song_diff | int | 1–10 | 演出曲目的内部难度值（来自曲目数据库） |
| venue_difficulty_modifier | venue_mod | float | 0.5–2.0 | 场地定义的难度加成 |

**Output Range:** 0.5 到 20（难度值）
**Example:** 曲目难度 6（中等）, 校园开放舞台 modifier=1.0 → `venue_difficulty = 6 × 1.0 = 6`

**Note:** 演出判定系统在此基础上叠加乐队技能、熟练度、士气等因素计算最终演出分数。内容数据库只提供 `venue_difficulty` 作为基准输入。

### 3. 配合技触发判定 (Combo Trigger Check)

The combo trigger check is defined as:

`combo_triggered = required_members_present AND trigger_condition_met`

**Where:**

`required_members_present = all(member in rehearsal_band for member in combo.required_members)`

`trigger_condition_met` depends on `combo.trigger_condition`:
- `REHEARSAL`: `true` when the combo members are in the same rehearsal session
- `PERFORMANCE`: `true` when the combo members are in the same performance
- `ALWAYS`: `true` whenever the required members are both in the band

**Example:** 倔强节奏 combo 需要林野 + 大伟。当排练计划包含这两人时 → `combo_triggered = true`

### 4. 观众容量收益上限公式 (Audience Capacity Cap)

The audience capacity reputation cap formula is defined as:

`reputation_cap = audience_capacity × 0.2`

**Variables:**
| Variable | Symbol | Type | Range | Description |
|----------|--------|------|-------|-------------|
| audience_capacity | capacity | int | 10–100000 | 场地观众容量 |

**Output Range:** 2 到 20000
**Example:** 校园开放舞台 capacity=50 → `reputation_cap = 50 × 0.2 = 10`。即使演出评分极高，单次演出声誉收益不超过 10。

**Purpose:** 防止低容量场地因偶然高分产生不合理的高收益，驱动玩家解锁更大场地。

## Edge Cases

- **If 解锁条件中的前置内容已被移除或不存在**：条件自动判定为 `false`，内容保持锁定。日志记录警告。
- **If 配合技所需角色中有角色离队或不可用**（如未来系统允许角色离队）：配合技自动失效，`combo_triggered = false`。
- **If 多个配合技同时满足触发条件**：所有配合技效果**叠加**（乘算）。例如两个 +15% 的配合技同时触发 → 总加成 = 1.15 × 1.15 = 1.3225（+32.25%）。
- **If 演出场地被解锁但当前没有可演出曲目**（所有曲目熟练度 < 60%）：场地仍显示为已解锁，但 UI 提示"需要先熟练一首曲目才能演出"。
- **If 声誉收益计算结果超过 audience_capacity 上限**：硬封顶至 `reputation_cap`，溢出部分丢弃。
- **If 剧情章节的关联事件中有事件尚未实现**：章节仍可解锁，但显示"故事待续"占位文本。未实现的事件不阻塞章节完成判定。
- **If `is_hidden = true` 的内容解锁条件意外满足**：内容立即从隐藏状态变为可见，并触发解锁提示（惊喜感设计）。
- **If 玩家在已完成的剧情章节推进后卸载并重装游戏**：通过存档/加载系统恢复进度。内容数据库本身不处理持久化。

## Tuning Knobs

| 参数名 | 当前值 | 安全范围 | 过低效果 | 过高效果 | 关联系统 |
|--------|--------|----------|----------|----------|----------|
| `venue_difficulty_modifier_range` | 0.5–2.0 | 0.3–3.0 | 场地间难度差异过小，缺乏成长感 | 低层级场地几乎不可能成功，玩家挫败 | 演出判定 |
| `reputation_reward_base_range` | 5–100 | 3–200 | 声誉成长过慢，解锁内容节奏拖沓 | 声誉成长过快，进度解锁失去意义 | 进度解锁 |
| `audience_capacity_cap_multiplier` | 0.2 | 0.1–0.5 | 小场地收益上限过低，玩家不愿演出 | 小场地收益上限过高，大场地失去吸引力 | 声誉系统 |
| `combo_effect_range` | +0.10–+0.30 | +0.05–+0.50 | 配合技效果微弱，玩家无视角色搭配 | 配合技效果过强，不触发配合技的组合无人问津 | 放置排练 |
| `chapter_completion_reputation` | 10–50 | 5–100 | 章节奖励微不足道，叙事失去动力 | 章节奖励过高，玩家为奖励而跳过文本 | 叙事事件 |
| `hidden_content_reveal_chance` | 100% | 50–100% | 隐藏内容可能永远不显示 | 无影响 | UI |

## Visual/Audio Requirements

- **场地卡片**：每个场地的展示卡片像一张演出海报，从旧到新的风格递进——校园开放舞台像用马克笔手绘的传单，音乐节像专业印刷的海报。锁定场地的海报被揉成纸团或浸湿模糊。
- **配合技触发特效**：当"倔强节奏"触发时，排练界面闪过一道铁锈红色的光效，像吉他弦和鼓槌碰撞出的火花。特效持续 0.5 秒，不干扰操作。
- **章节解锁动画**：新剧情章节解锁时，UI 像翻开日记的一页，纸张有泛黄和咖啡渍的质感。伴随轻柔的吉他扫弦声。
- **完成标记**：已完成的内容（章节、配合技最大化）显示一枚生锈的徽章，像从旧夹克上摘下来的纪念章。

## UI Requirements

- **内容总览界面**：一个"世界地图"式的界面，显示乐队当前的位置和可解锁的内容节点。已解锁的节点亮着暖黄灯光，锁定的节点是灰暗的剪影，隐藏节点完全不显示。
- **场地详情面板**：显示场地海报、难度信息、观众容量、历史最佳演出分数、解锁条件进度。
- **配合技图鉴**：已发现的配合技以卡片形式展示，包含角色头像组合、效果说明、触发次数统计。未发现的配合技显示为问号卡片，提示"尝试不同的角色组合"。
- **章节选择器**：像翻阅一本旧日记，左侧是章节列表，右侧是当前章节的叙事事件摘要和完成进度。
- **解锁提示**：内容解锁时，主界面显示对应类型的浮动提示——场地解锁显示舞台灯光闪烁，配合技解锁显示角色头像碰撞特效，章节解锁显示日记翻页动画。

## Acceptance Criteria

- **GIVEN** 校园开放舞台 `reputation_reward_base = 10`，**WHEN** 演出评分及格（1.0x），**THEN** 声誉奖励为 10
- **GIVEN** 校园开放舞台 `audience_capacity = 50`，**WHEN** 演出完美（2.0x）且计算声誉为 25，**THEN** 声誉收益硬封顶至 10（50 × 0.2）
- **GIVEN** 曲目难度 6，校园开放舞台 `difficulty_modifier = 1.0`，**WHEN** 计算场地难度，**THEN** 结果为 6
- **GIVEN** 排练计划包含林野和大伟，**WHEN** 判定配合技触发，**THEN** "倔强节奏"触发（+15% 技能成长）
- **GIVEN** 排练计划只包含林野，**WHEN** 判定配合技触发，**THEN** 无配合技触发
- **GIVEN** 两个 +15% 的配合技同时触发，**WHEN** 计算总加成，**THEN** 总加成 = 1.15 × 1.15 = 1.3225
- **GIVEN** 《地下室之光》熟练度 ≥ 40% 且声誉 ≥ 5，**WHEN** 判定校园开放舞台解锁条件，**THEN** 解锁成功
- **GIVEN** `chapter_first_show` 所有关联事件已完成，**WHEN** 判定章节完成状态，**THEN** 标记为已完成并发放奖励
- **GIVEN** `is_hidden = true` 的内容解锁条件意外满足，**WHEN** 系统检测解锁，**THEN** 内容从隐藏变为可见并触发惊喜提示
- **GIVEN** 剧情章节有关联事件尚未实现，**WHEN** 玩家查看章节进度，**THEN** 显示"故事待续"，不阻塞其他已完成内容

## Open Questions

- **Q1**: 是否需要"装饰/自定义"内容类型（如排练室装饰、角色服装）？MVP 不需要，但完整版可能是重要的成长感来源。
- **Q2**: 配合技的效果数值（+15%）是否需要随着角色关系深化而提升？这会将配合技与关系矩阵系统深度绑定。
- **Q3**: 场地解锁后是否可以有"升级"机制（如校园开放舞台升级后观众容量从 50 提升到 100）？还是每个场地固定参数？
- **Q4**: 隐藏内容 (`is_hidden = true`) 的比例应该多少？太多隐藏内容会让玩家困惑，太少则失去惊喜感。
- **Q5**: 剧情章节的内容文本应该存储在内容数据库中，还是独立的叙事内容系统？当前设计将 `linked_events` 放在内容数据库，但事件文本本身在叙事事件系统。
