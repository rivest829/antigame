# 叙事事件系统 (Narrative Events System)

> **Status**: Designed (pending review)
> **Author**: [user + agents]
> **Last Updated**: 2026-04-28
> **Implements Pillar**: 人物是活的, 选择是有意义的

## Overview

叙事事件系统是《地下排练室》的**运行时故事引擎**，负责将事件模板系统定义的静态模板转化为动态的游戏体验。它决定"什么时候触发事件、推送给哪个玩家、处理玩家的选择、应用选择的后果"。如果说事件模板系统是"剧本库"，叙事事件系统就是"导演+舞台"——它让剧本在舞台上活起来。

**系统边界**：
- **包含**：事件触发调度、事件队列管理、事件状态生命周期、玩家选择处理、后果应用、事件历史记录
- **不包含**：事件模板的具体定义（属于事件模板系统）、事件的UI展示（属于叙事事件UI）、角色基础状态存储（属于角色属性系统）、数值计算的具体公式（属于各业务系统）

**玩家如何感知**：
- 离线回来，看到主界面红点"2条新消息" → "我不在的时候发生了一些事"
- 打开事件，看到林野和大伟的争执 → "他们是真实的人，有真实的冲突"
- 做出选择，看到后果在接下来几天慢慢显现 → "我的选择有分量"
- 事件历史里翻看过去的记录 → "我们一起经历了这么多"

**为什么必须存在**：
没有叙事事件系统，事件模板只是死数据——模板定义了"什么条件下可能触发什么事件"，但谁来检查条件？谁来把事件推给玩家？谁来处理玩家的选择？叙事事件系统是连接"静态剧本"和"动态体验"的桥梁，是"人物是活的"这一支柱的核心执行层。

## Player Fantasy

> **你离线了4小时，回来打开游戏。主界面右上角有一个红点："2条新消息"。**
>
> 你点开第一条——是一张皱巴巴的纸条，上面用大伟潦草的字迹写着："林野今天又把《锈》的solo改了，我说原来的版本更有感觉，他说我根本不懂音乐。我们吵了20分钟。"
>
> 你有三个选择：
> 1. "支持林野：创新是乐队的灵魂"
> 2. "支持大伟：经典版本确实更有味道"
> 3. "各打五十大板：你们俩都冷静一下"
>
> 你选了第三条。纸条烧掉，取而代之的是一行淡灰色的字："两天后，林野主动找大伟道歉了。两人的关系没有变差，但也没有变好。"
>
> 你点开第二条——是一个小信封，里面是一片干枯的银杏叶。"排练室窗外的那棵树又掉叶子了。林野把它捡起来，夹在了谱架里。"纯叙事，没有选择。你只是看着这片叶子，想起上次登录时看到的那场争吵。
>
> **核心情感时刻**：当你发现你选择"各打五十大板"后，林野和大伟的关系值没有变化，但两人在接下来几次排练中触发了"倔强节奏"的概率提高了——因为那次争吵让他们更珍惜彼此的配合了。这不是写在任何文本里的，是系统根据你的选择悄悄调整的概率参数。你永远不会知道这个细节，但你会感觉到：**这个世界回应了我的选择。**

**这不是"读小说"的幻想——这是"参与一个正在发生的故事"的幻想。**

## Detailed Design

### Core Rules

#### 1. 事件触发时机

叙事事件系统在以下时机检查事件触发：

| 触发时机 | 检查频率 | 触发类型 | 说明 |
|----------|---------|---------|------|
| **离线收益结算后** | 每次登录时 | 全部类型 | 基于离线期间的角色状态变化判定 |
| **实时排练阶段性结算后** | 每5分钟游戏时间 | REHEARSAL, RELATIONSHIP | 排练过程中可能触发即时事件 |
| **演出结束后** | 每次演出完成 | PERFORMANCE, MILESTONE | 演出结果驱动的事件 |
| **玩家手动刷新** | 玩家操作 | 全部类型 | 点击"查看消息"时额外检查一次 |

**触发流程**：
```
时机到达
  → 读取所有事件模板
  → 对每个模板执行条件判定（调用事件模板系统的条件语法）
  → 收集所有满足条件的模板
  → 按优先级排序（MILESTONE > DAILY > REHEARSAL > RELATIONSHIP > PERFORMANCE）
  → 同优先级按 base_chance 降序
  → 依次尝试触发，直到达到 events_per_session 上限
```

#### 2. 事件队列管理

**事件队列结构**：
```
EventQueue:
  active_events: []        # 已触发、待处理的事件
  archived_events: []      # 已处理或已归档的事件（仅保留摘要）
  max_active: 5            # 活跃事件上限
```

**队列操作规则**：
- 新事件触发时，如果 `active_events.length < max_active`：直接加入队列
- 如果 `active_events.length >= max_active`：
  - 找到优先级最低的非MILESTONE事件
  - 将其归档（保留标题摘要）
  - 新事件加入队列
- MILESTONE事件永远不被归档（它们驱动主线进度）

**队列排序**：
活跃事件按以下规则排序（决定显示顺序）：
1. 优先级高 → 低
2. MILESTONE类型优先
3. 创建时间新 → 旧

#### 3. 事件状态生命周期

| 状态 | 进入条件 | 离开条件 | 玩家可见 |
|------|---------|---------|---------|
| **草稿 (Draft)** | 模板条件满足，生成实例 | 推送给玩家 | 不可见 |
| **活跃-未读 (Active_Unread)** | 推送到玩家队列 | 玩家打开事件界面 | 主界面红点 + 通知数 |
| **活跃-已读未处理 (Active_Read)** | 玩家打开事件 | 玩家做出选择 | 事件详情界面 |
| **已处理 (Resolved)** | 玩家做出选择 | 后果应用完成 | 事件历史记录（完整） |
| **已归档 (Archived)** | 超过堆积上限或手动归档 | 无（终态） | 事件历史记录（仅摘要） |

```
条件满足 → 草稿
  → 推送 → 活跃-未读（红点）
    → 玩家打开 → 活跃-已读未处理
      → 玩家选择 → 已处理 → 后果应用 → 归档
```

#### 4. 玩家选择处理

**分支选项展示**：
- 事件UI显示所有分支选项
- 每个选项检查 `option_condition`：
  - 满足：正常显示，可点击
  - 不满足：灰色显示，不可点击，hover显示原因

**选择处理流程**：
```
玩家点击选项
  → 记录选择
  → 应用立即后果（delay=0）
  → 如果有延迟后果：加入调度队列
  → 如果有连锁事件：加入事件队列（delay_hours后触发）
  → 标记事件为"已处理"
  → 通知叙事事件UI显示后果动画
  → 触发下游系统更新（角色属性、关系矩阵、声誉等）
```

#### 5. 后果应用

后果由事件模板系统定义，叙事事件系统负责执行：

| 后果类型 | 执行方式 | 目标系统 |
|----------|---------|---------|
| `STAT_CHANGE` | 立即调用角色属性系统API | 角色属性系统 |
| `SKILL_CHANGE` | 立即调用角色属性系统API | 角色属性系统 |
| `RELATIONSHIP_CHANGE` | 立即调用关系矩阵系统API | 关系矩阵系统 |
| `CHEMISTRY_CHANGE` | 立即调用化学反应系统API | 化学反应系统 |
| `UNLOCK_CONTENT` | 调用进度解锁系统API | 进度解锁系统 |
| `TRIGGER_EVENT` | 加入延迟调度队列 | 本系统 |
| `SET_FLAG` | 写入全局/角色标志存储 | 存档/加载系统 |

**延迟后果调度**：
```
ScheduledConsequence:
  - execute_at: timestamp
  - consequence_type: enum
  - parameters: dict
  - source_event_id: string

每小时检查一次调度队列，执行到期的后果。
```

#### 6. 事件历史记录

系统维护最近50条事件记录：

```json
{
  "event_history": [
    {
      "event_id": "evt_001",
      "template_id": "rel_argument_rehearsal",
      "title": "林野和大伟的争执",
      "type": "RELATIONSHIP",
      "timestamp": 1713800000,
      "selected_branch": "让两人冷静一下",
      "summary": "林野和大伟因solo版本争吵，玩家选择调解",
      "consequences": ["关系值不变", "倔强节奏触发概率+5%"]
    }
  ]
}
```

**历史用途**：
- 玩家回顾"我们经历了什么"
- 叙事事件引用（"记得那次争吵吗？"）
- 调试和数据分析

### States and Transitions

叙事事件系统作为运行时服务，其内部状态主要是事件实例的状态（见上文"事件状态生命周期"）。系统本身有以下运行模式：

| 模式 | 触发条件 | 行为 |
|------|---------|------|
| **待机 (Idle)** | 无触发时机 | 维护调度队列，检查到期延迟后果 |
| **触发检查 (Checking)** | 时机到达 | 遍历模板，执行条件判定 |
| **推送 (Pushing)** | 有新事件触发 | 加入队列，通知UI更新红点 |
| **处理中 (Resolving)** | 玩家做出选择 | 应用后果，更新状态 |

```
待机 → 触发检查 → 推送 → 待机
              ↓
         处理中 → 待机
```

## Formulas

### 1. 事件触发频率控制 (Event Trigger Rate)

```
events_per_session = min(max_events_per_session, base_event_rate × session_duration_hours × mood_modifier)
```

**Variables:**
| Variable | Symbol | Type | Range | Description |
|----------|--------|------|-------|-------------|
| max_events_per_session | max | int | 3-5 | 单次会话最大事件数 |
| base_event_rate | rate | float | 0.3-0.5 | 每小时基础事件率 |
| session_duration_hours | duration | float | 0-∞ | 离线时长或在线会话时长 |
| mood_modifier | mood | float | 0.5-1.5 | 心情修正（全队平均心情<30时=0.5，>70时=1.5） |

**Output Range:** 0 to max_events_per_session
**Example:** max=4, rate=0.3, duration=4h, mood=1.0 → `events = min(4, 0.3 × 4 × 1.0) = min(4, 1.2) = 1`

**MVP简化**：每小时最多触发1个事件，单次登录最多3个。

---

### 2. 事件优先级评分 (Event Priority Score)

用于决定多个事件同时满足时的触发顺序：

```
priority_score = template_priority × 100 + type_bonus + freshness_bonus
```

**Type Bonus:**
| 类型 | Bonus |
|------|-------|
| MILESTONE | 1000 |
| DAILY | 0 |
| REHEARSAL | 100 |
| RELATIONSHIP | 200 |
| PERFORMANCE | 150 |

**Freshness Bonus:**
```
freshness_bonus = max(0, 50 - hours_since_last_trigger)
```
- 同一模板24小时内未触发 → +50
- 同一模板刚触发过 → +0

**Example:** `rel_argument_rehearsal` (priority=5, type=RELATIONSHIP, last triggered 10h ago)
→ `score = 5 × 100 + 200 + 40 = 740`

---

### 3. 延迟后果调度 (Delayed Consequence Scheduling)

```
execute_at = current_timestamp + (delay_hours × 3600)
```

**调度检查**：
```
FOR EACH scheduled IN consequence_queue:
    IF current_timestamp >= scheduled.execute_at:
        apply_consequence(scheduled)
        remove_from_queue(scheduled)
```

**检查频率**：每小时一次（与离线收益计算同步）。

---

### 4. 事件堆积衰减 (Event Pileup Decay)

当玩家长时间未登录，旧事件需要衰减以保持体验新鲜：

```
IF hours_since_last_login > 48:
    max_active_events = max(3, original_max - floor((hours - 48) / 24))
    # 每多离线24小时，堆积上限减1，最低3
```

**设计意图**：离线3天回来，不应该面对10个未读事件。事件堆积上限随离线时间递减，迫使系统只保留最重要的事件。

---

## Edge Cases

- **If 事件堆积超过 max_active_events 且所有事件都是 MILESTONE**：保留最新的3个MILESTONE，其余归档。MILESTONE事件不应被丢弃。
- **If 玩家在事件界面做出选择后游戏崩溃**：选择结果通过存档系统保存。重启后，事件状态为"已处理"，后果继续应用流程。
- **If 延迟后果执行时目标角色已不存在**（如极端情况）：后果静默失败，记录错误日志。
- **If 连锁事件的 next_event 条件在 delay_hours 后不再满足**：连锁事件以草稿状态进入队列，触发检查时条件不满足 → 不触发，静默丢弃。
- **If 同一模板在短时间内多次满足条件**（如1小时内）：受事件模板系统的 cooldown_hours 限制，不会重复触发。
- **If 事件后果导致角色状态超出边界**（如心情>100）：由目标系统（角色属性系统）负责截断，本系统只发出变化请求。
- **If 玩家在事件未读状态下离线**：事件保持"活跃-未读"状态，下次登录时继续显示红点。
- **If 事件UI正在展示时新事件触发**：新事件加入队列，当前事件处理完成后自动展示下一个（如果有）。
- **If 事件模板系统在运行时更新**（如热更新）：已生成的事件实例不受影响，新触发的事件使用更新后的模板。

## Tuning Knobs

| 参数 | 当前值 | 安全范围 | 调整影响 | 关联 |
|------|--------|---------|---------|------|
| `max_events_per_session` | 3 | 2-5 | 单次登录最多看到几个事件。越高=故事感越强，但可能信息过载 | 事件频率 |
| `base_event_rate` | 0.3 | 0.1-0.5 | 每小时基础事件触发率。越高=世界越"活跃" | 事件频率 |
| `mood_modifier_low` | 0.5 | 0.3-0.8 | 全队心情<30时的触发率修正。越低=心情不好时事件越少 | 事件频率 |
| `mood_modifier_high` | 1.5 | 1.2-2.0 | 全队心情>70时的触发率修正。越高=心情好时事件越多 | 事件频率 |
| `event_pileup_max` | 5 | 3-10 | 活跃事件堆积上限。越高=玩家可能错过的事件越少，但处理压力大 | 队列管理 |
| `pileup_decay_rate` | 1/24h | 0.5-2/24h | 离线堆积衰减速度。越高=长时间离线回来看到的事件越少 | 队列管理 |
| `milestone_protected` | true | true/false | MILESTONE事件是否受堆积限制保护。false=可能丢失主线事件 | 主线进度 |
| `delay_consequence_check_interval_minutes` | 60 | 30-120 | 延迟后果检查频率（分钟）。越短=延迟后果越准时 | 调度 |
| `history_max_entries` | 50 | 20-100 | 事件历史最大条目数。越高=可追溯的历史越长 | 存档 |

**调优策略**：
- 如果玩家反馈"回来总是没事发生"：提高 `base_event_rate` 或降低 `event_pileup_max`
- 如果玩家反馈"事件太多看不过来"：降低 `max_events_per_session` 或提高 `event_pileup_max`
- 如果长时间离线玩家感到困惑"怎么才1个事件"：降低 `pileup_decay_rate`

## Acceptance Criteria

### 功能测试

- **GIVEN** 离线4小时且条件满足，**WHEN** 登录游戏，**THEN** 主界面显示未读事件红点（1-3个）
- **GIVEN** `rel_argument_rehearsal` 模板条件满足，**WHEN** 触发检查，**THEN** 事件正确加入活跃队列
- **GIVEN** 活跃事件数已达5个，**WHEN** 新事件触发，**THEN** 最早的一个非MILESTONE事件被归档
- **GIVEN** 玩家打开事件界面，**WHEN** 点击某分支选项，**THEN** 选择被记录，立即后果应用，事件标记为已处理
- **GIVEN** 某分支有 `delay_hours=24` 的延迟后果，**WHEN** 24小时后，**THEN** 延迟后果正确执行
- **GIVEN** 事件状态为"活跃-未读"，**WHEN** 游戏关闭并重启，**THEN** 事件状态保持"活跃-未读"
- **GIVEN** MILESTONE事件触发且活跃队列已满，**WHEN** 检查堆积处理，**THEN** MILESTONE不被归档，保留在队列中

### 边界测试

- **GIVEN** 离线72小时，**WHEN** 登录，**THEN** 堆积上限衰减生效（最多3个活跃事件）
- **GIVEN** 连锁事件的 next_event 在 delay 后条件不再满足，**WHEN** 检查触发，**THEN** 连锁事件静默丢弃
- **GIVEN** 玩家在事件界面选择后游戏崩溃，**WHEN** 重启，**THEN** 事件为"已处理"状态，后果不重复应用
- **GIVEN** 事件后果使角色心情>100，**WHEN** 应用后果，**THEN** 角色属性系统正确截断到100

### 集成测试

- **GIVEN** 放置排练完成，**WHEN** 排练后触发检查，**THEN** 正确检查REHEARSAL类型模板
- **GIVEN** 演出判定为"出色"，**WHEN** 演出后触发检查，**THEN** 正确检查PERFORMANCE类型模板
- **GIVEN** 声誉达到阈值，**WHEN** 进度解锁系统检查，**THEN** 正确接收解锁请求
- **GIVEN** 事件处理完成，**WHEN** 存档，**THEN** 事件状态和历史正确保存

### 性能测试

- **GIVEN** 50个模板同时检查，**WHEN** 执行触发判定，**THEN** 耗时 < 10ms
- **GIVEN** 事件队列有5个活跃事件，**WHEN** 排序，**THEN** 耗时 < 1ms
- **GIVEN** 历史记录50条，**WHEN** 查询，**THEN** 耗时 < 5ms

## Open Questions

- **Q1**: 事件触发是否应该考虑"玩家上次选择"？例如玩家总是选择"调解"，后续是否减少冲突类事件？这会增加个性化，但也可能导致玩家"训练"系统。
- **Q2**: 延迟后果的 `delay_hours` 是真实时间还是游戏时间？当前设计为真实时间（与放置机制一致），但是否需要支持"游戏内天数"？
- **Q3**: 事件历史是否需要支持搜索/筛选？MVP 只支持滚动查看，但搜索功能对长期玩家很有价值。
- **Q4**: 如果玩家在事件处理中途被强制中断（如来电、系统杀后台），事件状态如何恢复？当前设计依赖存档系统，但需要在UI层增加"恢复中断会话"的机制。
- **Q5**: 是否需要"事件回放"功能——允许玩家重新观看已处理事件的完整内容（包括分支选项）？这对沉浸感有帮助，但增加存档数据量。
