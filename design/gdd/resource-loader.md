# 资源加载系统 (Resource Loader)

> **Status**: Designed (pending review)
> **Author**: [user + agents]
> **Last Updated**: 2026-04-23
> **Implements Pillar**: 节奏是轻松的

## Overview

资源加载系统是《地下排练室》的**动态资源管理中枢**，负责游戏运行时所有外部资源（图片、音频、场景、数据文件）的异步加载、缓存、释放和错误处理。它将资源加载从"阻塞等待"转变为"后台预载+按需取用"，确保玩家在任何场景切换或内容展示时都不会遇到突兀的加载卡顿。

**系统边界**：
- **包含**：资源加载队列、缓存管理、异步加载接口、加载进度追踪、资源释放策略、错误回退机制
- **不包含**：具体的资源文件（属于美术/音频资产）、资源导入设置（属于 Godot 导入系统）、资源打包/热更新（属于构建系统）

**玩家如何感知**：
- 玩家**理想情况下不感知**——资源在需要之前已静默加载完成
- 当资源尚未加载时，显示风格统一的占位图/占位音频，而非空白或卡顿
- 加载进度以符合垃圾摇滚美学的方式呈现（如一张逐渐被墨水填满的废纸）

**为什么必须存在**：
没有资源加载系统，场景切换时会出现明显的卡顿——图片逐个弹出、音频延迟播放。这直接破坏"节奏是轻松的"支柱。移动端的存储和内存限制也要求我们主动管理资源生命周期，避免内存溢出。

## Player Fantasy

资源加载系统没有直接的玩家幻想——玩家感受到的是它消除的负面体验：

> 你点击"查看角色详情"，林野的头像和立绘瞬间出现——不是从模糊到清晰的渐进加载，而是像翻开相册时照片已经贴在那里。你听不到任何卡顿，感受不到任何等待。这个系统做得越好，你越不会意识到它的存在。

**核心情感**：**"无 friction"**。玩家不应该思考"这张图加载完了吗"——当他们想看的时候，内容已经在那里了。

**参考体验**：
- 《星露谷物语》的资源加载——几乎无感知，所有内容似乎都已经在那里
- 《塞尔达传说：旷野之息》的异步加载——你在奔跑时，远处的地形在后台悄悄加载

## Detailed Design

### Core Rules

#### 1. 资源分类体系

所有资源按类型和加载策略分类：

| 资源类型 | 文件扩展名 | 加载策略 | 缓存策略 | MVP 预估总量 |
|----------|-----------|----------|----------|-------------|
| **纹理 (Texture)** | .png, .jpg, .webp | 异步加载 | LRU 缓存 | ~20-30 张 |
| **音频 (Audio)** | .ogg, .mp3, .wav | 异步加载 | 常驻缓存 | ~10-15 个 |
| **场景 (Scene)** | .tscn | 按需加载 | 不缓存 | ~8 个 |
| **数据 (Data)** | .json, .yaml | 启动时预载 | 常驻内存 | ~5 个文件 |
| **字体 (Font)** | .ttf, .otf | 启动时预载 | 常驻内存 | ~2-3 个 |
| **Shader** | .gdshader | 启动时预载 | 常驻内存 | ~2-3 个 |

#### 2. 加载模式

系统支持三种加载模式：

| 加载模式 | 说明 | 使用场景 | 阻塞性 |
|----------|------|----------|--------|
| **PRELOAD** | 游戏启动时批量加载 | 核心数据、字体、常用 UI 元素 | 阻塞（有进度条） |
| **BACKGROUND** | 后台静默加载 | 预见到玩家即将需要的资源 | 非阻塞 |
| **ONDEMAND** | 即时请求，异步加载 | 玩家触发时才需要的资源 | 非阻塞（有占位符） |

#### 3. 缓存管理

**缓存层级**：

```
L1: 常驻缓存 (Resident Cache)
  - 内容：核心数据、字体、当前场景资源
  - 策略：永不自动释放
  - 上限：无（由内容量自然限制）

L2: LRU 缓存 (LRU Cache)
  - 内容：纹理、音频
  - 策略：最近最少使用
  - 上限：texture_cache_size_mb / audio_cache_size_mb

L3: 无缓存 (No Cache)
  - 内容：场景文件
  - 策略：场景卸载时立即释放
```

**LRU 缓存淘汰**：
- 当缓存达到上限时，移除最久未使用的资源
- 被淘汰的资源如果再次需要，重新从磁盘加载
- 资源引用计数 > 0 时不淘汰（防止正在使用的资源被释放）

#### 4. 异步加载接口

```gdscript
# 请求加载资源（异步）
ResourceLoader.load_request(resource_path, priority, callback)

# 检查资源是否已缓存
ResourceLoader.is_cached(resource_path) -> bool

# 获取已缓存资源（立即返回，无加载）
ResourceLoader.get_cached(resource_path) -> Resource

# 预加载一批资源（后台）
ResourceLoader.preload_batch(resource_paths, progress_callback)

# 释放指定资源
ResourceLoader.release(resource_path)

# 清空 LRU 缓存
ResourceLoader.clear_lru_cache()
```

**加载回调**：
```gdscript
func on_resource_loaded(resource_path: String, status: LoadStatus, resource: Resource):
    # status: LOADING / LOADED / FAILED / CANCELLED
    # resource: 加载成功的资源对象（失败时为 null）
```

#### 5. 加载优先级

```gdscript
enum LoadPriority {
    CRITICAL = 0,    # 立即加载，阻塞其他请求
    HIGH = 1,        # 尽快加载（当前场景所需）
    NORMAL = 2,      # 标准队列
    LOW = 3,         # 后台预载
    IDLE = 4         # 仅空闲时加载
}
```

**优先级动态调整**：
- 资源请求时分配初始优先级
- 如果资源在视口中变为可见，优先级提升至 HIGH
- 如果资源在视口中变为不可见，优先级降低至 LOW

#### 6. 占位符系统

当资源尚未加载完成时，显示占位符：

| 资源类型 | 占位符 | 说明 |
|----------|--------|------|
| 纹理 | 纯色块 + 资源名称 | 脏黄色 (#8B7355)，手写风格文字 "加载中..." |
| 音频 | 静音 | 无声音，不报错 |
| 场景 | 转场动画延长 | 延长当前转场动画直到场景加载完成 |
| 数据 | 默认值 | 使用硬编码默认值，后台加载完成后刷新 |

**占位符升级**：资源加载完成后，占位符平滑过渡为实际资源（0.2 秒淡入）。

#### 7. 错误处理与回退

| 错误类型 | 检测方式 | 回退策略 |
|----------|----------|----------|
| 文件不存在 | 文件系统检查 | 使用默认占位资源，记录错误日志 |
| 文件损坏 | 解析失败 | 使用默认占位资源，记录错误日志 |
| 加载超时 | 超时计时器（5秒） | 取消加载，使用占位资源，记录警告 |
| 内存不足 | 内存监控 | 触发 LRU 缓存清理，重试加载 |
| 格式不支持 | 文件头检查 | 拒绝加载，使用占位资源 |

#### 8. 内存预算管理

**移动端内存预算**（来自 technical-preferences.md）：
- 总内存上限：256 MB
- 资源加载系统预算：64 MB（纹理 + 音频缓存）

**内存监控**：
```gdscript
# 每 30 秒检查一次内存使用
func _on_memory_check():
    var used_mb = OS.get_static_memory_usage() / 1024 / 1024
    if used_mb > memory_warning_threshold:
        # 内存警告：清理 LRU 缓存
        clear_lru_cache()
    if used_mb > memory_critical_threshold:
        # 内存临界：强制释放所有非必要资源
        clear_all_caches()
```

**阈值设置**：
| 阈值 | 值 | 行为 |
|------|-----|------|
| memory_warning_threshold | 192 MB (75%) | 清理 LRU 缓存 |
| memory_critical_threshold | 224 MB (87.5%) | 强制释放所有缓存 |

### States and Transitions

资源加载系统的操作状态：

| 状态 | 说明 | 触发条件 |
|------|------|----------|
| **IDLE** | 无加载任务 | 队列为空 |
| **LOADING** | 正在加载资源 | 队列中有任务 |
| **PRELOADING** | 启动预加载进行中 | 游戏启动时 |
| **PAUSED** | 加载暂停 | 游戏进入后台 |

```
游戏启动
    │
    ▼
┌─────────────┐    预加载完成      ┌─────────┐    加载请求      ┌─────────┐
│ PRELOADING  │ ─────────────────► │  IDLE   │ ──────────────► │ LOADING │
└─────────────┘                    └────┬────┘                 └────┬────┘
                                        │                           │
                                        │ 队列清空                    │ 队列清空
                                        │◄──────────────────────────┘
                                        │
                              游戏进入后台 │ 游戏回到前台
                                        ▼                           │
                                   ┌─────────┐◄─────────────────────┘
                                   │ PAUSED  │
                                   └─────────┘
```

**并发加载限制**：
- 同时加载的资源数上限：3 个（防止 I/O 过载）
- 超出限制的请求进入队列，按优先级排序

## Formulas

### 1. 缓存大小估算公式 (Cache Size Estimation)

The cache size estimation formula is defined as:

`cache_size_mb = texture_count × avg_texture_size_mb + audio_count × avg_audio_size_mb`

**Variables:**
| Variable | Symbol | Type | Range | Description |
|----------|--------|------|-------|-------------|
| texture_count | tex_n | int | 10-50 | 同时缓存的纹理数量 |
| avg_texture_size_mb | tex_sz | float | 0.1-2.0 | 单张纹理平均大小（MB） |
| audio_count | aud_n | int | 5-20 | 同时缓存的音频数量 |
| avg_audio_size_mb | aud_sz | float | 0.5-3.0 | 单个音频平均大小（MB） |

**Output Range:** 2 MB 到 100 MB
**Example:** 20 张纹理（平均 0.5MB）+ 10 个音频（平均 1MB）→ `cache = 20 × 0.5 + 10 × 1 = 20 MB`

### 2. 加载队列优先级排序公式 (Queue Priority)

Resources in the loading queue are sorted by:

`sort_score = priority_value × 1000 - time_in_queue_ms × 0.001`

**Variables:**
| Variable | Symbol | Type | Range | Description |
|----------|--------|------|-------|-------------|
| priority_value | prio | int | 0-4 | 加载优先级（CRITICAL=0, HIGH=1, NORMAL=2, LOW=3, IDLE=4） |
| time_in_queue_ms | time | int | ≥0 | 在队列中等待的时间（毫秒） |

**Output Range:** -∞ 到 4000（数值越小，优先级越高）
**Example:** HIGH 优先级（1），已等待 5000ms → `score = 1 × 1000 - 5000 × 0.001 = 1000 - 5 = 995`

**说明**：优先级值权重远高于等待时间——一个 LOW 优先级的资源即使等待很久，也不会超过 NORMAL 优先级的新请求。这防止了低优先级资源饿死高优先级资源。

### 3. 纹理内存占用公式 (Texture Memory)

The texture memory usage formula is defined as:

`texture_memory_bytes = width × height × bytes_per_pixel`

**Variables:**
| Variable | Symbol | Type | Range | Description |
|----------|--------|------|-------|-------------|
| width | w | int | 1-2048 | 纹理宽度（像素） |
| height | h | int | 1-2048 | 纹理高度（像素） |
| bytes_per_pixel | bpp | int | 4 | Godot 默认压缩格式每像素字节数 |

**Output Range:** 4 bytes 到 ~16 MB (2048×2048×4)
**Example:** 512×512 纹理 → `memory = 512 × 512 × 4 = 1,048,576 bytes = 1 MB`

**MVP 纹理预算**：
| 用途 | 尺寸 | 数量 | 总内存 |
|------|------|------|--------|
| 角色立绘 | 512×512 | 2 | 2 MB |
| UI 元素 | 256×256 | 15 | ~4 MB |
| 场景背景 | 1024×1024 | 3 | 12 MB |
| 图标 | 64×64 | 20 | ~0.3 MB |
| **合计** | — | — | **~18 MB** |

## Edge Cases

- **If 请求加载的资源已在缓存中**：立即返回缓存对象，不重新加载。
- **If 同一资源被多次同时请求**：只执行一次加载，所有请求者共享结果。加载完成后通知所有等待的回调。
- **If 资源加载失败且没有占位资源**：创建一个纯色纹理（脏黄色 #8B7355）作为紧急占位，确保 UI 不崩溃。
- **If 内存使用超过 critical_threshold**：强制释放所有 LRU 缓存资源，取消所有 BACKGROUND/IDLE 优先级的加载请求。保留 CRITICAL/HIGH 优先级的加载。
- **If 资源文件在运行时被外部修改**（如热更新）：Godot 的 ResourceLoader 会检测到文件变化，下次加载时自动重新读取。
- **If 加载队列中有 100+ 个请求**：触发队列长度警告，丢弃 IDLE 优先级的请求。防止内存或队列溢出。
- **If 资源加载回调中再次请求加载同一资源**：防止递归死锁——如果检测到循环依赖，返回错误并记录日志。
- **If 游戏进入后台时有未完成的加载任务**：暂停所有加载，游戏回到前台后恢复。
- **If 网络资源加载超时**（MVP 不涉及网络资源，但预留）：重试 3 次，每次间隔 2 秒，全部失败后使用占位资源。
- **If 纹理尺寸超过设备支持的最大值**（如 2048×2048）：Godot 自动缩放，记录警告。

## Tuning Knobs

| 参数名 | 当前值 | 安全范围 | 过低效果 | 过高效果 | 关联 |
|--------|--------|----------|----------|----------|------|
| `texture_cache_size_mb` | 32 | 16-64 | 纹理频繁重新加载，卡顿 | 内存占用过高，可能导致 OOM | 内存 |
| `audio_cache_size_mb` | 16 | 8-32 | 音频频繁重新加载，延迟 | 内存占用过高 | 内存 |
| `max_concurrent_loads` | 3 | 1-5 | 加载队列过长，延迟高 | I/O 竞争，加载效率下降 | 性能 |
| `load_timeout_seconds` | 5 | 3-10 | 正常加载被误判超时 | 卡死的加载占用队列太久 | 稳定性 |
| `memory_warning_threshold_mb` | 192 | 128-224 | 内存警告太晚，可能 OOM | 过早清理缓存，加载频繁 | 内存 |
| `memory_critical_threshold_mb` | 224 | 160-240 | 临界反应太晚 | 过于激进，正常游戏被影响 | 内存 |
| `placeholder_fade_duration` | 0.2 | 0.1-0.5 | 替换太突兀 | 替换太慢，玩家注意到 | 视觉 |
| `lru_check_interval_seconds` | 30 | 10-60 | 内存问题发现太晚 | 过于频繁，CPU 开销 | 性能 |
| `queue_max_length` | 100 | 50-200 | 请求容易丢弃 | 队列过长，内存占用 | 稳定性 |

## Visual/Audio Requirements

资源加载系统**不直接定义视觉资产**，但定义了加载状态的视觉表现：

**加载占位符**：
- 纹理占位：脏黄色 (#8B7355) 纯色矩形，中央手写风格文字 "..." 或资源名称缩写
- 进度指示：加载大量资源时，显示一张逐渐被墨水晕染的废纸——墨迹扩散表示加载进度

**加载完成过渡**：
- 占位符到实际资源的过渡：0.2 秒淡入（opacity 0→1）
- 不使用缩放或位移动画——保持简洁，不分散玩家注意力

## Acceptance Criteria

- **GIVEN** 资源已在缓存中，**WHEN** 请求加载该资源，**THEN** 立即返回缓存对象，不重新加载
- **GIVEN** 同一资源被同时请求 3 次，**WHEN** 加载完成，**THEN** 只执行 1 次加载，3 个请求者都收到回调
- **GIVEN** 纹理缓存已达 32MB 上限，**WHEN** 加载新纹理，**THEN** 淘汰最久未使用的纹理
- **GIVEN** 资源文件不存在，**WHEN** 尝试加载，**THEN** 显示占位资源，记录错误日志
- **GIVEN** 资源加载超过 5 秒，**WHEN** 超时触发，**THEN** 取消加载，显示占位资源
- **GIVEN** 内存使用超过 192MB，**WHEN** 内存检查触发，**THEN** 清理 LRU 缓存
- **GIVEN** 内存使用超过 224MB，**WHEN** 内存检查触发，**THEN** 强制释放所有缓存，取消 BACKGROUND/IDLE 加载
- **GIVEN** 游戏进入后台时有未完成的加载，**WHEN** 游戏回到前台，**THEN** 恢复加载，状态正确
- **GIVEN** 加载队列中有 100 个请求，**WHEN** 新请求进入，**THEN** 丢弃 IDLE 优先级请求
- **GIVEN** 512×512 纹理加载完成，**WHEN** 计算内存占用，**THEN** 占用 1MB（512×512×4 bytes）
- **GIVEN** HIGH 优先级资源和低优先级资源同时在队列，**WHEN** 排序加载顺序，**THEN** HIGH 优先级先加载

## Open Questions

- **Q1**: MVP 是否需要在启动时预加载所有资源？还是按需加载即可？预加载增加启动时间但减少运行时卡顿。
- **Q2**: 是否需要实现资源的"流式加载"（如大背景图分块加载）？MVP 资源量小，可能不需要。
- **Q3**: 纹理压缩格式如何选择？Godot 支持多种压缩格式，不同格式影响质量和内存占用。
- **Q4**: 是否需要支持资源的"懒加载"——只加载视口内可见的资源？这对滚动列表类 UI 有意义。
- **Q5**: 是否需要实现资源的引用计数自动释放？还是完全依赖 LRU 策略？引用计数更精确但增加复杂度。
