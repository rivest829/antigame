# 演出判定系统 (Performance Judgment System)

> **Status**: In Design
> **Author**: [user + agents]
> **Last Updated**: 2026-04-27
> **Implements Pillar**: 选择是有意义的, 成长是可见的, 节奏是轻松的

## Overview

演出判定系统是《地下排练室》的**核心闭环仲裁者**，负责将乐队的排练成果转化为具体的演出结果。它是"设置→等待→收获"循环中的"收获"环节——玩家前期的所有决策（选谁排练、练什么歌、选什么场地、是否触发化学反应）在这里汇总，通过一套清晰的判定规则产生一个可感知的结果。没有演出判定，排练就是无意义的数值积累；有了它，每一次排练都有了方向。

**系统边界**：
- **包含**：演出评分计算、演出成败判定、评分等级划分、演出结果数据结构、演出反馈生成
- **不包含**：难度计算（属于演出难度系统）、声誉收益计算（属于声誉系统）、解锁判定（属于进度解锁系统）、演出视觉表现（属于演出结果UI）、演出过程中的实时操作（本游戏无实时操作，是纯放置）

**玩家如何感知**：
- 离线回来，看到"演出结果"弹窗 → "我们的演出有结果了"
- 评分从"灾难"到"完美"的梯度 → "从失败到成功的完整光谱"
- 具体的评分分解（技巧/配合/台风/观众反应） → "知道哪里好，哪里不好"
- 失败演出的温和处理 → "失败不会毁掉一切，只是没有进步"

**为什么必须存在**：
没有演出判定，放置排练的成长没有出口。玩家会问"我练了这么多，然后呢？"演出判定提供了"然后"——它是"成长是可见的"支柱的最终呈现，也是"选择是有意义的"的终极验证：你的选择不仅影响了排练效率，还影响了最终演出结果。

## Player Fantasy

> **"你不在现场，但你在场。"**

你打开游戏，离线了 6 小时。屏幕上没有直接弹出结果——而是先播放了一段 5 秒钟的"演出回想"：模糊的地下室画面，你听到《锈》的失真吉他声，然后是观众的反应（或稀稀落落，或热烈）。然后，结果界面浮现。

你看到了什么？

**如果成功**：
- 技巧 72 分：林野的 solo 有几个小瑕疵，但整体稳健
- 配合 65 分：大伟的鼓和林野的吉他在第二段有些脱节
- 台风 58 分：两个人都太紧张了，没怎么看观众
- 观众反应："还不错，可以再来"
- 总评：**成功** — 声誉 +12

你盯着"配合 65 分"看了一会儿。你知道原因——你们只排练了 3 次《锈》，熟练度才 55%。你想着：如果再多练几次，触发了"倔强节奏"，这个分数会更高。

**如果失败**：
- 技巧 45 分：林野弹错了几个和弦
- 配合 38 分：鼓和吉他完全不合拍
- 台风 40 分：两个人都像在考试
- 观众反应："嗯……加油"
- 总评：**平庸** — 声誉 +3（幸好不是 0）

你没有感到挫败。你感到的是：**我知道为什么。** 你点了详情，系统告诉你"熟练度不足 + 士气偏低"。你看了看大伟的士气——才 35。你想起三天前那场争吵事件。你明白了：不是运气不好，是你没有处理好乐队的状态就安排了演出。

**核心情感时刻**：
当你把《锈》的熟练度练到 80%，林野和大伟的关系值到了 75，触发了"倔强节奏"，然后选择了校园开放舞台（难度适中）——演出结果弹出"完美"两个字。

那一刻不是"我赢了"——而是**"我的判断是对的"**。你从 6 小时前的决策中获得了正反馈。这个反馈不是随机的，是你应得的。

**参考体验**：
- 《足球经理》的比赛结果——不是你操作的，但你的准备决定了结果
- 《赛马娘》的比赛——训练成果在比赛中的具象化

**这不是"我必须赢"的幻想——这是"我的准备配得上结果"的幻想。**

## Detailed Design

### Core Rules

#### 1. 演出评分维度 (Performance Dimensions)

演出从 4 个维度评分，每个维度 0-100 分：

| 维度 | 权重 | 决定因素 | 玩家感知 |
|------|------|---------|---------|
| **技巧 (Technique)** | 30% | 角色乐器技能 vs 曲目要求 | "他们弹得准不准" |
| **配合 (Synergy)** | 30% | 角色关系值、化学反应效果、熟练度 | "他们像一支乐队还是像两个人" |
| **台风 (Stage Presence)** | 20% | 角色士气、状态值 | "他们在台上自不自在" |
| **观众反应 (Audience Response)** | 20% | 场地容量、声誉、随机因子 | "观众买不买账" |

**总分计算公式**：
```
total_score = technique × 0.30 + synergy × 0.30 + stage_presence × 0.20 + audience_response × 0.20
```

#### 2. 各维度评分计算

**技巧 (Technique)**：
```
technique = min(100, average(skill_coverage) × proficiency_bonus × instrument_match_bonus)
```

- `skill_coverage`：每个参与角色的技能值 / 曲目要求值，取平均
- `proficiency_bonus`：1.0 + (熟练度 / 100) × 0.5 → 熟练度 100% 时 +50%
- `instrument_match_bonus`：所有所需乐器都被覆盖 = 1.0，有未覆盖 = 0.7

**配合 (Synergy)**：
```
synergy = min(100, relationship_base × chemistry_bonus × proficiency_synergy)
```

- `relationship_base`：参与角色平均关系值
- `chemistry_bonus`：1.0 + 活跃化学反应效果总和（如 +15% = 1.15）
- `proficiency_synergy`：1.0 + (熟练度 / 100) × 0.3

**台风 (Stage Presence)**：
```
stage_presence = min(100, average_morale × state_factor)
```

- `average_morale`：参与角色平均士气值
- `state_factor`：
  - 所有角色精力 ≥ 50：1.0
  - 有角色精力 20-50：0.85
  - 有角色精力 < 20：0.6（但这种情况不允许安排演出）

**观众反应 (Audience Response)**：
```
audience_response = min(100, base_audience_score × reputation_factor × venue_fit × random_factor)
```

- `base_audience_score`：场地容量决定的基础分（小场地 60，大场地 40——大场地更难讨好）
- `reputation_factor`：1.0 + (reputation / max_reputation) × 0.2 → 高声誉观众更宽容
- `venue_fit`：乐队能力与场地 tier 的匹配度。匹配 = 1.0，差距大 = 0.7
- `random_factor`：0.9 - 1.1 的随机波动（代表现场不可控因素）

#### 3. 演出评级 (Performance Rating)

根据总分划分 5 个评级：

| 评级 | 分数范围 | 名称 | 声誉收益 | 解锁影响 | 玩家感知 |
|------|---------|------|---------|---------|---------|
| S | 90-100 | 完美 (Perfect) | 2× 基础 | 可能触发特殊解锁 | "这是属于我们的时刻" |
| A | 75-89 | 出色 (Great) | 1.5× 基础 | 正常解锁检查 | "比预期还要好" |
| B | 60-74 | 成功 (Success) | 1.0× 基础 | 正常解锁检查 | "稳住了" |
| C | 40-59 | 平庸 (Mediocre) | 0.5× 基础 | 无解锁 | "能听，但没什么印象" |
| F | 0-39 | 灾难 (Disaster) | 0 | 无解锁，可能有负面事件 | "我们还是别谈这场了" |

#### 4. 难度与评级的动态关系

演出难度系统提供的 `final_difficulty` 影响评分：

```
difficulty_penalty = final_difficulty / 100 × 0.3  # 最高 30% 评分削减
difficulty_bonus = max(0, (50 - final_difficulty) / 100 × 0.2)  # 低难度无奖励

adjusted_total = total_score × (1.0 - difficulty_penalty + difficulty_bonus)
```

**设计意图**：高难度演出天然更难获得高分。在 Livehouse 获得"成功"比在校园舞台获得"成功"更有价值。

#### 5. 失败保护机制

**灾难评级不会直接造成声誉损失**：
- 声誉不会减少（只会不增加）
- 但可能触发负面叙事事件（如"演出失败后乐队士气低落"）
- 如果连续 3 次灾难：触发特殊事件"我们是否该解散？"（可选择继续或重置）

**为什么这样设计**：
放置游戏的核心是"轻松"。如果一次失败的演出让玩家失去 hours of progress，会破坏"节奏是轻松的"支柱。失败的机会成本是时间，不是倒退。

### States and Transitions

演出判定系统在演出流程中的阶段：

| 阶段 | 触发条件 | 系统行为 |
|------|---------|---------|
| **准备** | 玩家安排演出计划 | 锁定演出参数（成员、曲目、场地），计算预览难度 |
| **等待** | 玩家离线或等待演出时间 | 无行为，但角色状态可能变化 |
| **判定** | 演出时间到达 | 基于当前实际状态计算 4 维度评分 |
| **评级** | 评分计算完成 | 根据总分确定评级 |
| **结算** | 评级确定 | 计算声誉收益、技能成长、熟练度成长 |
| **反馈** | 结算完成 | 生成反馈文本、通知其他系统 |

## Formulas

### 1. 总分计算公式 (Total Score)

```
total_score = technique × 0.30 + synergy × 0.30 + stage_presence × 0.20 + audience_response × 0.20
```

**Adjusted Total (after difficulty)**:
```
difficulty_penalty = final_difficulty / 100 × 0.3
difficulty_bonus = max(0, (50 - final_difficulty) / 100 × 0.2)
adjusted_total = total_score × (1.0 - difficulty_penalty + difficulty_bonus)
```

**Output Range:** 0 to 100
**Example:** technique=70, synergy=65, stage=60, audience=55
→ `total = 70×0.3 + 65×0.3 + 60×0.2 + 55×0.2 = 21 + 19.5 + 12 + 11 = 63.5`
→ difficulty=40 → `penalty=0.12`, `bonus=0.02`
→ `adjusted = 63.5 × 0.90 = 57.15` → **平庸**

---

### 2. 技巧评分公式 (Technique)

```
technique = min(100, average(skill_coverage) × proficiency_bonus × instrument_match_bonus)
```

**Sub-formulas:**
```
skill_coverage = actual_skill / required_skill  for each required instrument
average(skill_coverage) = sum(coverage) / count(instruments)
proficiency_bonus = 1.0 + (proficiency / 100) × 0.5
instrument_match_bonus = 1.0 if all covered else 0.7
```

**Variables:**
| Variable | Symbol | Type | Range | Description |
|----------|--------|------|-------|-------------|
| actual_skill | act | int | 0-100 | 角色实际乐器技能 |
| required_skill | req | int | 1-100 | 曲目要求的乐器技能 |
| proficiency | prof | float | 0-100 | 乐队对该曲目的熟练度 |

**Output Range:** 0 to 100
**Example:** 吉他要求 40，林野吉他 48 → coverage=1.2；鼓要求 25，大伟鼓 28 → coverage=1.12；avg=1.16
→ proficiency=55% → bonus=1.275；all covered → match=1.0
→ `technique = min(100, 1.16 × 1.275 × 1.0) = min(100, 1.479) = 100` (capped)

---

### 3. 配合评分公式 (Synergy)

```
synergy = min(100, relationship_base × chemistry_bonus × proficiency_synergy)
```

**Sub-formulas:**
```
relationship_base = average_relationship_score / 100  # normalize to 0-1
chemistry_bonus = 1.0 + sum(active_chemistry_effects)
proficiency_synergy = 1.0 + (proficiency / 100) × 0.3
```

**Output Range:** 0 to 100
**Example:** relationship=65 → base=0.65；stubborn_rhythm active → bonus=1.15；proficiency=55% → synergy=1.165
→ `synergy = min(100, 0.65 × 1.15 × 1.165) = min(100, 0.871) = 87.1`

---

### 4. 台风评分公式 (Stage Presence)

```
stage_presence = min(100, average_morale × state_factor)
```

**State Factor:**
| Condition | Value |
|-----------|-------|
| All energy ≥ 50 | 1.0 |
| Any energy 20-50 | 0.85 |
| Any energy < 20 | 0.6 (should not happen) |

**Output Range:** 0 to 100
**Example:** morale=70 → `presence = min(100, 70 × 1.0) = 70`

---

### 5. 观众反应公式 (Audience Response)

```
audience_response = min(100, base_audience_score × reputation_factor × venue_fit × random_factor)
```

**Sub-formulas:**
```
base_audience_score = 60 - (venue_tier - 1) × 5  # tier 1=60, tier 5=40
reputation_factor = 1.0 + (reputation / max_reputation) × 0.2
venue_fit = 1.0 - abs(band_tier - venue_tier) × 0.1  # band_tier estimated from skills
random_factor = randf_range(0.9, 1.1)
```

**Output Range:** 0 to 100
**Example:** venue tier 2 → base=55；reputation=50, max=200 → factor=1.05；band_tier=2, venue_tier=2 → fit=1.0；random=0.95
→ `response = min(100, 55 × 1.05 × 1.0 × 0.95) = min(100, 54.9) = 54.9`

## Edge Cases

- **If 角色在演出前状态剧变（如士气暴跌到 0）**: 以实际演出时的状态计算。玩家会看到"预览难度 vs 实际难度"不同。如果状态极差导致无法演出（如所有角色精力 < 20），演出自动取消，显示"乐队状态太差，演出延期"。
- **If 曲目熟练度为 0%**: `proficiency_bonus = 1.0`，没有加成但也不惩罚。技巧分主要由技能值决定。
- **If 化学反应效果到期但演出已开始**: 以演出开始时的状态快照为准。已到期效果不影响本次演出。
- **If random_factor 极端值（0.9 或 1.1）**: 可能导致同一准备水平产生不同结果。这是设计意图——现场演出有不可控因素。但 random 只影响 20% 的 audience_response，对总分影响有限（±2 分）。
- **If 连续 3 次灾难**: 触发特殊事件"我们是否该解散？"，提供选择：继续（士气 -20）或重置（回到上次存档）。这是防止玩家陷入死亡螺旋的保护机制。
- **If 演出时乐队缺少角色（如某人生病）**: 如果曲目所需乐器无人覆盖，instrument_match_bonus = 0.7，技巧分大幅下降。如果关键角色缺失（如主唱不在），可能直接无法演出。
- **If 多个曲目演出（setlist）**: 每首歌独立计算评分，最终总分为加权平均（按曲目长度或难度加权）。MVP 只支持单曲演出，Post-MVP 扩展 setlist。

| 来源系统 | 输入数据 | 作用 |

### 依赖顺序

```
角色属性系统 ──┐
```

| `audience_weight` | 0.20 | 0.15-0.30 | 观众反应在总分中的占比 | 总分 |
| `proficiency_technique_bonus` | 0.50 | 0.30-0.70 | 熟练度对技巧分的加成幅度 | 技巧 |
| `proficiency_synergy_bonus` | 0.30 | 0.20-0.50 | 熟练度对配合分的加成幅度 | 配合 |
| `difficulty_penalty_max` | 0.30 | 0.20-0.40 | 最高难度对评分的最大削减 | 难度调整 |
| `difficulty_bonus_max` | 0.20 | 0.10-0.30 | 低难度对评分的最大奖励 | 难度调整 |
| `random_range` | 0.9-1.1 | 0.85-1.15 | 随机因子范围 | 观众反应 |
| `base_audience_tier_1` | 60 | 55-65 | 最小场地的观众基础分 | 观众反应 |
| `disaster_streak_threshold` | 3 | 2-5 | 连续灾难触发特殊事件的次数 | 失败保护 |

**调优策略**：
- 如果玩家反馈"演出结果太随机"：缩小 `random_range` 或降低 `audience_weight`
- 如果玩家反馈"熟练度不重要"：提高 `proficiency_technique_bonus`
- 如果玩家反馈"高难度演出不可能成功"：降低 `difficulty_penalty_max`
- 如果玩家从不失败感到无聊：提高 `difficulty_penalty_max` 或降低各维度基础分

## Visual/Audio Requirements

### 视觉需求

**演出回想动画（开场）**：
- 5 秒模糊画面：场地背景 + 乐器剪影
- 评分越高，画面越清晰、色调越暖
- 评分越低，画面越暗、有"噪点"效果

**评分展示动画**：
- 4 个维度分数从 0 滚动到实际值
- 滚动速度：0.5 秒/维度
- 每个维度有自己的颜色条：技巧=蓝色，配合=绿色，台风=紫色，观众=橙色

**评级揭晓**：
- 总分计算完成后，评级大字弹出
- 完美：金色 + 轻微震动 + 粒子效果
- 出色：银色 + 缩放效果
- 成功：白色 + 淡入
- 平庸：灰色 + 静态
- 灾难：暗红 + 屏幕轻微"碎裂"效果

**反馈文本区域**：
- 评级下方显示 3 行反馈：亮点、改进建议、观众引语
- 引语用引号包裹，字体略小，颜色较淡

### 音频需求

**回想阶段**：
- 曲目音频的 5 秒片段（失真、现场感）
- 评分越高，音频越清晰；越低，越嘈杂

**评分滚动音效**：
- 数字滚动时的"咔嗒"声（类似老式计数器）

**评级揭晓音效**：
- 完美：完整和弦 + 观众欢呼
- 出色：简短和弦 + 掌声
- 成功：单音符 + 轻微掌声
- 平庸：静音或低沉的"嗯"
- 灾难：不和谐音符 + 嘘声

**反馈阶段**：
- 观众引语播放时，有对应的背景音（掌声、聊天声、或寂静）

## UI Requirements

### 演出结果界面

**布局（从上到下）**：
1. **标题**："演出结果" + 场地名称
2. **回想区域**：5 秒动画（可跳过）
3. **评级区域**：大字评级 + 总分
4. **评分分解**：4 个维度的分数条
5. **收益区域**：声誉 +X、技能成长、熟练度成长
6. **反馈区域**：亮点、改进建议、观众引语
7. **按钮**："继续" / "查看详情" / "分享"

**评分条设计**：
- 水平条，0-100 刻度
- 填充动画从 0 滚动到实际值
- 颜色分级：
  - 0-39：红色
  - 40-59：橙色
  - 60-74：黄色
  - 75-89：浅绿
  - 90-100：深绿

**详情面板**：
- 点击"查看详情"后展开
- 显示每个维度的详细计算过程
- 例如技巧分："技能覆盖率 116% × 熟练度加成 127% × 乐器匹配 100% = 100（封顶）"
- 帮助玩家理解"为什么是这个分数"

### 演出历史界面

- 时间线显示最近 20 次演出
- 每场显示：日期、场地、曲目、评级、总分
- 筛选：按场地 / 按曲目 / 按评级
- 点击某场可查看完整结果

## Acceptance Criteria

### 功能测试

- [ ] 4 个维度评分各自在 0-100 范围
- [ ] 总分 = 维度加权平均
- [ ] 难度惩罚正确应用（高难度降低总分）
- [ ] 评级正确映射到分数区间
- [ ] 完美演出获得 2× 声誉
- [ ] 灾难演出获得 0 声誉
- [ ] 化学反应效果正确影响配合分
- [ ] 熟练度正确影响技巧分和配合分
- [ ] 观众反应包含随机因子

### 边界测试

- [ ] 所有维度 100 分 → 总分 100（封顶）
- [ ] 所有维度 0 分 → 总分 0
- [ ] 难度 100 → 最高 30% 惩罚
- [ ] 连续 3 次灾难 → 触发解散事件
- [ ] 角色状态极差 → 演出取消或分数极低

### 集成测试

- [ ] 从演出难度系统接收 difficulty → 正确调整总分
- [ ] 从化学反应系统接收效果 → 正确影响配合分
- [ ] 输出评级到声誉系统 → 正确计算收益
- [ ] 输出结果到进度解锁系统 → 正确检查解锁
- [ ] 输出结果到存档系统 → 正确保存历史

### 性能测试

- [ ] 演出判定计算耗时 < 5ms
- [ ] 结果界面渲染耗时 < 50ms

## Open Questions

- **演出过程是否需要可视化？** 当前是纯放置——玩家离线后回来直接看结果。是否需要在演出进行时显示一个"直播中"的画面（即使玩家不参与）？
- **Setlist（多曲目演出）如何在 MVP 处理？** 当前设计假设单曲演出。MVP 是否只支持单曲？
- **观众反应引语是否需要大量文本？** MVP 每个评级需要 3-5 条引语，共 15-25 条。是否需要更多变体？
- **失败保护机制是否足够温和？** 连续 3 次灾难才触发特殊事件，是否太宽松？或者应该更严格以防止玩家滥用？
- **技能成长和熟练度成长是否在演出后一次性给予？** 还是分散在排练过程中？当前设计是演出后一次性结算。
- **是否需要"演出回放"功能？** 即允许玩家重新观看演出的"模拟过程"。这会增加开发量，但增强沉浸感。
