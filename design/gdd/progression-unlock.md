# 进度解锁系统 (Progression Unlock System)

> **Status**: Designed (pending review)
> **Author**: [user + agents]
> **Last Updated**: 2026-04-29
> **Implements Pillar**: 成长是可见的

## Overview

进度解锁系统是《地下排练室》的**成长感引擎**，负责检查玩家的乐队是否满足了解锁新内容的条件，并在满足时执行解锁、通知玩家。它是"成长是可见的"这一支柱最直接的表现形式——每次解锁都是一个"叮"的正反馈时刻，让玩家明确感知到"我们进步了"。

**系统边界**：
- **包含**：解锁条件定义、解锁条件判定、解锁执行、解锁通知、已解锁内容管理
- **不包含**：解锁内容的具体数据定义（属于内容数据库）、解锁内容的UI展示（属于各UI系统）、解锁条件的来源数据（属于各业务系统）

**玩家如何感知**：
- 熟练度达到60%，屏幕闪过一道光 → "解锁新曲目《404》"
- 声誉达到21点，收到一条消息 → "解锁校园开放舞台"
- 林野和大伟的关系值达到60，弹出提示 → "解锁新化学反应：倔强节奏"
- 主界面上原本灰色的图标亮起 → "又多了一些可以探索的东西"

**为什么必须存在**：
没有进度解锁系统，玩家的成长是分散的、不可感的——技能涨了、熟练度升了、声誉增加了，但这些都只是数字变化。进度解锁系统把这些分散的成就编织成"里程碑"，让玩家每一次登录都有期待、每一次进步都有奖励。它是放置游戏留存的核心驱动力之一。

## Detailed Design

### Core Rules

#### 1. 解锁条件类型

进度解锁系统支持以下类型的解锁条件：

| 条件类型 | 说明 | 示例 |
|----------|------|------|
| `REPUTATION_THRESHOLD` | 声誉值达到阈值 | 声誉 ≥ 21 解锁校园舞台 |
| `PROFICIENCY_THRESHOLD` | 某曲目熟练度达到阈值 | 《锈》熟练度 ≥ 60% 解锁《404》 |
| `RELATIONSHIP_THRESHOLD` | 某对角色关系值达到阈值 | 林野-大伟 ≥ 60 解锁"倔强节奏" |
| `EVENT_COMPLETED` | 完成特定叙事事件 | 完成"深夜的排练室"事件 |
| `PERFORMANCE_SUCCESS` | 在某场地成功演出N次 | 在校园舞台成功演出3次 |
| `COMBO_TRIGGERED` | 触发特定配合技 | 首次触发"倔强节奏" |
| `MULTI_CONDITION` | 多个条件同时满足 | 声誉≥30 AND 《锈》熟练度≥80% |

**条件组合逻辑**：支持 `AND` 和 `OR` 组合。`AND` 表示所有条件必须同时满足，`OR` 表示任一条件满足即可。

#### 2. 解锁内容类型

| 内容类型 | 说明 | MVP 数量 |
|----------|------|----------|
| `SONG` | 新曲目 | 3-5 |
| `VENUE` | 新演出场地 | 1-2 |
| `CHEMISTRY` | 新化学反应 | 1-2 |
| `STORY_CHAPTER` | 剧情章节 | 2-3 |
| `UI_FEATURE` | UI功能（如历史记录标签页） | 1-2 |
| `CHARACTER` | 新角色 | 0（MVP只有林野+大伟） |

#### 3. 解锁检查时机

系统在以下时机执行解锁检查：

| 时机 | 触发来源 | 检查范围 |
|------|---------|---------|
| **排练结束后** | 放置排练系统 | 熟练度相关解锁 |
| **演出结束后** | 演出判定系统 | 声誉/演出相关解锁 |
| **事件处理后** | 叙事事件系统 | 事件/关系相关解锁 |
| **化学反应触发后** | 化学反应系统 | 配合技相关解锁 |
| **玩家手动刷新** | 玩家操作 | 全部解锁 |
| **登录时** | 存档加载后 | 全部解锁（防止遗漏） |

**检查流程**：
```
时机到达
  → 读取所有未解锁内容
  → 对每个内容检查其解锁条件
  → 收集所有满足条件的内容
  → 依次执行解锁
  → 生成解锁通知
  → 通知主界面UI显示
```

#### 4. 解锁执行流程

```
解锁条件满足
  → 标记内容状态为"已解锁"
  → 记录解锁时间戳
  → 记录解锁来源（哪个系统触发的检查）
  → 生成解锁通知
  → 如果有连锁解锁（解锁A后B的条件也满足）：递归检查
  → 通知相关UI系统更新
```

**连锁解锁示例**：
- 解锁校园舞台 → 检查"校园演出成功3次"的条件（当前0/3）→ 不满足
- 解锁《404》 → 检查"《404》熟练度≥60%解锁《地下室之光》" → 不满足
- 解锁"倔强节奏" → 检查"触发倔强节奏3次解锁巅峰反应" → 不满足

#### 5. 解锁通知机制

**单次解锁**：
- 主界面显示弹窗：内容图标 + 内容名称 + 简短描述
- 弹窗持续时间：3秒（可点击跳过）
- 弹窗消失后，对应UI元素从灰色变为彩色

**批量解锁**（离线期间或连锁触发多个）：
- 合并为一个摘要弹窗："你有 X 个新解锁！"
- 点击后展开列表，逐个展示
- 最多一次显示5个，超出部分分批显示

**通知优先级**（同时触发时）：
1. 新角色/剧情章节（最高）
2. 新场地
3. 新曲目
4. 新化学反应
5. UI功能（最低）

#### 6. 已解锁内容管理

```json
{
  "unlocked_content": {
    "songs": ["rust", "basement_light", "404"],
    "venues": ["basement", "campus_stage"],
    "chemistries": ["daily_sync", "stubborn_rhythm"],
    "story_chapters": ["ch_01", "ch_02"],
    "ui_features": ["history_tab", "performance_history"]
  },
  "unlock_history": [
    {
      "content_id": "404",
      "content_type": "SONG",
      "unlocked_at": 1713800000,
      "triggered_by": "proficiency_check",
      "source_value": 60
    }
  ]
}
```

**重复解锁保护**：已解锁的内容不可再次解锁。检查条件时自动跳过已解锁内容。

### States and Transitions

进度解锁系统本身是无状态的计算服务，但每个内容有以下状态：

| 状态 | 说明 | 转换条件 |
|------|------|---------|
| **锁定 (Locked)** | 条件未满足，玩家不可见或灰色显示 | 条件满足 → 已解锁 |
| **已解锁 (Unlocked)** | 条件已满足，玩家可见且可用 | 无（终态） |
| **新解锁 (Newly_Unlocked)** | 刚解锁，尚未被玩家查看 | 玩家查看后 → 已解锁 |

```
锁定 → 条件满足 → 新解锁（弹窗通知） → 玩家查看 → 已解锁
```

**"新解锁"状态的UI表现**：
- 主界面：对应图标有闪烁光效
- 列表页：对应条目有"NEW"标签
- 玩家首次点击后，"NEW"标签消失，状态变为"已解锁"

## Formulas

### 1. 解锁条件判定 (Unlock Condition Evaluation)

```
is_unlocked(content) = AND(evaluate(condition) for condition in content.conditions)
```

**单条件判定**：
```
evaluate(condition):
    actual_value = fetch_from_system(condition.source_system, condition.parameter)
    required_value = condition.threshold
    
    SWITCH condition.operator:
        case ">=": return actual_value >= required_value
        case "<=": return actual_value <= required_value
        case "==": return actual_value == required_value
        case "IN": return actual_value IN required_value_set
        case "EVENT_COMPLETED": return event_id IN completed_events
```

**Variables:**
| Variable | Symbol | Type | Range | Description |
|----------|--------|------|-------|-------------|
| actual_value | av | varies | varies | 从目标系统读取的实际值 |
| required_value | rv | varies | varies | 内容数据库定义的阈值 |
| operator | op | enum | >= / <= / == / IN / EVENT_COMPLETED | 条件运算符 |

**Example:** 解锁《404》的条件： proficiency("rust") >= 60
- actual_value = 62（来自曲目数据库）
- required_value = 60
- operator = ">="
- evaluate = 62 >= 60 = true → 解锁

### 2. 批量解锁处理 (Batch Unlock Processing)

```
unlock_batch = [content for content in all_content if is_unlocked(content) and not is_already_unlocked(content)]

IF len(unlock_batch) == 1:
    show_single_unlock_popup(unlock_batch[0])
ELSE IF len(unlock_batch) <= 5:
    show_batch_unlock_summary(unlock_batch)
ELSE:
    show_batch_unlock_summary(unlock_batch[0:5])
    schedule_remaining(unlock_batch[5:])  # 分批显示
```

**Variables:**
| Variable | Symbol | Type | Range | Description |
|----------|--------|------|-------|-------------|
| unlock_batch | batch | list | 0-N | 本次检查中满足条件的未解锁内容 |
| batch_display_max | max | int | 3-10 | 单次最多显示解锁数 |

---

### 3. 连锁解锁深度控制 (Chain Unlock Depth)

防止解锁A触发解锁B，B触发C，C触发D...无限递归：

```
chain_depth_max = 3

def process_unlock(content, depth=0):
    IF depth > chain_depth_max:
        log_warning("Chain unlock depth exceeded")
        return
    
    mark_unlocked(content)
    generate_notification(content)
    
    # 检查连锁解锁
    related_content = find_related_unlocks(content)
    FOR rc in related_content:
        IF is_unlocked(rc) and not is_already_unlocked(rc):
            process_unlock(rc, depth + 1)
```

## Edge Cases

- **If 多个条件同时满足导致大量解锁**：按优先级分批显示，每批最多5个，间隔2秒。避免弹窗轰炸。
- **If 解锁条件引用的内容ID不存在**：跳过该条件，记录警告日志。不影响其他条件判定。
- **If 解锁条件引用的系统尚未实现**（如角色系统未上线）：该条件永远返回false，内容保持锁定。系统上线后自动解锁（登录时全量检查）。
- **If 玩家离线期间满足大量解锁条件**：登录时一次性检查所有条件，按批量规则显示。
- **If 已解锁内容的数据后续更新**（如曲目难度调整）：解锁状态不变，新数据立即生效。玩家无需重新解锁。
- **If 解锁通知弹窗期间玩家切换界面**：弹窗不消失，悬浮显示。玩家返回主界面时继续展示。
- **If 玩家在同一时刻满足同一内容的多个解锁条件**（如同时满足声誉和熟练度条件）：只解锁一次，通知中显示"通过多种途径达成"。
- **If 解锁内容需要前置内容已解锁**（如解锁《404》需要先解锁《锈》）：前置条件已包含在条件定义中，正常检查即可。如果前置条件未满足，主条件也不满足。
- **If 存档中的已解锁列表与内容数据库不一致**（如删除了某内容）：忽略不一致项，记录警告。已解锁但不存在的条目静默处理。

## Tuning Knobs

| 参数 | 当前值 | 安全范围 | 调整影响 | 关联 |
|------|--------|---------|---------|------|
| `batch_display_max` | 5 | 3-10 | 单次最多显示解锁数。越高=一次性展示越多，但可能信息过载 | 通知 |
| `batch_interval_seconds` | 2 | 1-5 | 批量解锁分批显示的间隔。越短=展示越快 | 通知 |
| `popup_duration_seconds` | 3 | 2-5 | 单次解锁弹窗持续时间。越长=玩家越不容易错过 | 通知 |
| `chain_depth_max` | 3 | 2-5 | 连锁解锁最大递归深度。越高=连锁反应越复杂 | 连锁解锁 |
| `new_badge_duration_hours` | 24 | 12-72 | "NEW"标签显示时长。越长=玩家越不容易错过新内容 | UI标记 |
| `retroactive_unlock_check` | true | true/false | 是否回溯检查历史成就。true=系统上线后自动解锁已满足条件的内容 | 兼容性 |

**调优策略**：
- 如果玩家反馈"解锁弹窗太多太烦"：降低 `batch_display_max` 或 `popup_duration_seconds`
- 如果玩家反馈"经常错过新解锁"：提高 `popup_duration_seconds` 或 `new_badge_duration_hours`
- 如果连锁解锁导致一次性解锁10+内容：降低 `chain_depth_max`

## Visual/Audio Requirements

### 视觉需求

**单次解锁弹窗**：
- 背景：半透明黑色遮罩
- 中央：内容图标（大）+ 内容名称（涂鸦字体）+ 简短描述
- 动画：图标从屏幕中央放大弹出，伴随光效粒子
- 持续时间：3秒，然后自动淡出
- 风格：像一张刚被发现的演出传单，从虚空中飘落到屏幕中央

**批量解锁摘要**：
- 顶部："你有 X 个新解锁！"（大号手写字体）
- 下方：横向滚动的小图标列表
- 点击每个图标：展开详细信息
- 背景：深色，带噪点纹理

**新解锁标记**：
- 主界面图标：右上角红色"NEW"标签，闪烁光效
- 列表条目：左侧竖条高亮（颜色根据内容类型变化）
- 首次点击后标记消失

### 音频需求

**解锁音效**：
- 普通解锁：轻快的吉他扫弦（和弦）
- 重要解锁（场地/剧情）：电吉他强力和弦 + 钹击
- 批量解锁：一段上升的琶音（解锁数量越多，琶音越长）

## Acceptance Criteria

### 功能测试

- **GIVEN** 《锈》熟练度达到60%，**WHEN** 排练结束触发解锁检查，**THEN** 《404》正确解锁并显示弹窗
- **GIVEN** 声誉达到21点，**WHEN** 演出结束触发解锁检查，**THEN** 校园开放舞台正确解锁
- **GIVEN** 同时满足3个解锁条件，**WHEN** 触发检查，**THEN** 显示批量解锁摘要（3个图标）
- **GIVEN** 已解锁《404》，**WHEN** 再次触发解锁检查，**THEN** 《404》不重复解锁
- **GIVEN** 解锁连锁（解锁A后B的条件也满足），**WHEN** 处理A的解锁，**THEN** B也正确解锁

### 边界测试

- **GIVEN** 解锁条件引用的系统未实现，**WHEN** 检查条件，**THEN** 该条件返回false，内容保持锁定
- **GIVEN** 存档中的已解锁列表包含不存在的内容ID，**WHEN** 加载存档，**THEN** 静默忽略，记录警告
- **GIVEN** 离线期间满足10个解锁条件，**WHEN** 登录，**THEN** 分批显示（每批5个，间隔2秒）
- **GIVEN** 连锁解锁深度超过3层，**WHEN** 处理解锁，**THEN** 在第4层停止，记录警告

### 集成测试

- **GIVEN** 从声誉系统读取声誉值，**WHEN** 检查声誉条件，**THEN** 使用最新值
- **GIVEN** 从放置排练系统读取熟练度，**WHEN** 检查熟练度条件，**THEN** 使用最新值
- **GIVEN** 解锁新曲目，**WHEN** 打开排练计划UI，**THEN** 新曲目出现在选择列表中
- **GIVEN** 解锁新场地，**WHEN** 打开演出安排UI，**THEN** 新场地出现在选择列表中
- **GIVEN** 解锁状态变化，**WHEN** 存档，**THEN** 已解锁列表正确保存

### 性能测试

- **GIVEN** 50个可解锁内容，**WHEN** 全量检查，**THEN** 耗时 < 5ms
- **GIVEN** 20个已解锁内容，**WHEN** 加载存档，**THEN** 耗时 < 2ms

## Open Questions

- **Q1**: 是否需要"隐藏解锁"——即满足条件但玩家不知道的内容，需要探索发现？这会增加神秘感，但可能让玩家困惑。
- **Q2**: 解锁条件是否需要支持"累计"类型？例如"累计排练100小时解锁XX"，而不是"单次排练达到XX"。
- **Q3**: 已解锁内容是否可重新锁定？例如某些剧情选择导致"失去场地访问权"。当前设计为单向解锁。
- **Q4**: 是否需要"解锁预览"功能——玩家可以查看"还差多少解锁XX"？这对目标驱动有帮助，但可能降低探索感。
- **Q5**: 解锁通知是否需要支持"稍后查看"——玩家可以选择不立即查看批量解锁，而是稍后从历史记录中查看？
