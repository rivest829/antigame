# 曲目数据库 (Song Database)

> **Status**: Designed (pending review)
> **Author**: [user + agents]
> **Last Updated**: 2026-04-23
> **Implements Pillar**: 成长是可见的

## Overview

曲目数据库是《地下排练室》的**排练目标仓库**，定义了乐队可以排练的所有曲目的静态属性。它既是纯数据层（为放置排练系统提供每首曲目的难度、所需乐器、排练收益和解锁条件），也是玩家成长感的主要来源——玩家通过解锁新曲目和看到曲目熟练度提升，直观地感受到"乐队在进步"。

**系统边界**：
- **包含**：曲目的名称、难度等级、所需乐器与技能门槛、排练收益参数、解锁条件、熟练度成长曲线
- **不包含**：曲目熟练度的实时计算与存储（属于放置排练系统）、曲目的音频播放（属于音频系统）、曲目的演出评分逻辑（属于演出判定系统）

**玩家如何感知**：
- 解锁新曲目时看到"新增可排练：《XXX》" → "乐队能驾驭更多风格了"
- 曲目熟练度从生疏涨到精通 → "这首歌终于练熟了，下次演出可以选它"
- 高难度曲目需要特定技能门槛 → "主唱的嗓音还不够，得多练基本功"

**为什么必须存在**：
没有曲目数据库，放置排练系统就失去了目标——玩家不知道"练什么"、"练到什么程度"、"什么时候算准备好了"。曲目数据库将抽象的"技能成长"转化为具体的、可命名的、有情感附着的目标（"我们总算把《XXX》练下来了"），是实现"成长是可见的"和"选择是有意义的"两大支柱的核心载体。

## Player Fantasy

玩家应该感受到：**"这些歌不只是曲目——这是我们的声音，我们的故事。"**

你滑动曲目列表，看到的不是难度数字和乐器标签。你看到的是乐队的成长日记：

- 《地下室之光》是乐队成立第一天练的，现在看到它，你记得那天林野紧张得拨错了和弦
- 《锈》是主唱失恋后写的，每次排练它，大伟的节奏都会不自觉地变得温柔
- 《404》是那次差点解散的危机后，全员一起完成的——现在看到它，你知道这支乐队挺过来了

**核心情感时刻**：
演出前选曲。你最终选了《锈》，不是因为它的难度和当前乐队能力最匹配，而是因为"这首歌代表了我们"。演出成功了，观众喊安可，你看着列表想：**下一首写什么？**

那一刻你不是在优化数值——你是在**定义这支乐队的身份**。就像真实生活中，你永远不会说"我选了这首歌因为它的 BPM 最合适"——你说的是"这是我们的歌"。

**参考体验**：
- 《吉他英雄》的曲目列表——不是"游戏关卡"，而是"你会弹的歌"（但我们的曲目承载的是故事，不是操作技巧）
- 真实乐队的 setlist——每首歌的位置都有意义，开场曲要炸，压轴曲要让人记住，安可曲要让全场一起唱

**这不是"收集全曲目"的幻想**——这是"用歌单书写我们的历史"的幻想。

## Detailed Design

### Core Rules

**1. 曲目数据结构**

每个曲目条目包含以下字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `song_id` | 字符串 | 唯一标识符，如 `basement_light` |
| `name` | 字符串 | 显示名称，如《地下室之光》 |
| `description` | 文本 | 曲目背景故事（1–2 句话） |
| `genre_tags` | 字符串数组 | 风格标签，如 `["原创", "后朋"]` |
| `difficulty_label` | 枚举 | 离散难度标签 |
| `difficulty_value` | 整数 | 内部数值，1–10 |
| `required_instruments` | 字典 | `{乐器: 最低技能门槛}` |
| `use_default_threshold` | 布尔 | `true` 时使用默认映射（难度×10） |
| `skill_growth_per_hour` | 字典 | `{乐器: 每小时技能成长值}` |
| `base_proficiency_per_hour` | 浮点数 | 基础熟练度成长率（%/小时） |
| `proficiency_curve_type` | 枚举 | 成长曲线类型：线性/递减/阶梯 |
| `unlock_conditions` | 条件数组 | 解锁所需的组合条件 |
| `performance_ready_threshold` | 整数 | 可演出的最低熟练度，默认 60% |

**2. 难度系统**

| 离散标签 | 内部数值范围 | 默认技能门槛 |
|----------|-------------|-------------|
| 入门 | 1–2 | 10–20 |
| 简单 | 3–4 | 30–40 |
| 中等 | 5–6 | 50–60 |
| 困难 | 7–8 | 70–80 |
| 地狱 | 9–10 | 90–100 |

默认映射公式：`default_threshold = difficulty_value × 10`

每首歌可通过 `use_default_threshold = false` 覆盖默认值，实现非线性难度设计（如一首标签为"中等"但门槛特别高的曲目，作为隐藏挑战）。

**3. 熟练度系统**

每首曲目独立追踪熟练度，范围 0–100%（整数）。

| 等级标签 | 熟练度范围 | 效果 |
|----------|-----------|------|
| 生疏 | 0–20% | 排练效率 -20%，演出评分无加成 |
| 熟悉 | 21–40% | 排练效率 基准，演出评分 +5% |
| 熟练 | 41–60% | 排练效率 +10%，演出评分 +10% |
| 精通 | 61–80% | 排练效率 +20%，演出评分 +15% |
| 大师 | 81–100% | 排练效率 +30%，演出评分 +20% |

熟练度成长受曲线类型影响：
- **线性**：每小时成长固定（最慢但稳定）
- **递减**：初始成长快，越接近 100% 越慢（符合学习曲线）
- **阶梯**：在等级阈值处成长速度突变（营造"突破感"）

**4. 排练收益参数**

每首歌定义以下收益参数（由放置排练系统读取并应用）：

- `skill_growth_per_hour`：每乐器每小时技能成长值（基础值，受角色状态和熟练度修正）
- `base_proficiency_per_hour`：每小时熟练度成长百分比（基础值，受曲线修正）

歌曲不定义心情/精力变化——这些由放置排练系统根据排练时长和角色状态统一计算。

**5. 解锁条件系统**

解锁条件采用组合条件设计，支持 AND/OR 逻辑：

**条件类型**：

| 类型 | 参数 | 说明 |
|------|------|------|
| `SKILL_THRESHOLD` | `instrument`, `min_skill` | 某乐器技能 ≥ X |
| `SONG_PROFICIENCY` | `song_id`, `min_proficiency` | 某曲目熟练度 ≥ Y% |
| `REPUTATION_MILESTONE` | `min_reputation` | 声誉 ≥ Z（预留，声誉系统尚未设计） |
| `BAND_SIZE` | `min_members` | 乐队人数 ≥ N（预留） |

**组合逻辑**：
- 条件数组内所有条件默认以 **AND** 逻辑组合（必须全部满足）
- 每个条件可设置 `inverted: true` 实现 NOT 逻辑
- 条件数组可嵌套子数组实现 OR 逻辑

**6. MVP 曲目清单**

| 曲目 | 难度 | 所需乐器 | 技能门槛 | 排练收益 | 解锁条件 | 背景 |
|------|------|----------|----------|----------|----------|------|
| 《地下室之光》 | 入门 (2) | 吉他 | 吉他 ≥ 15 | 吉他 +3/hr, 熟练度 +5%/hr | 初始解锁 | 乐队成立第一天练的原创 |
| 《锈》 | 简单 (4) | 吉他、鼓 | 吉他 ≥ 30, 鼓 ≥ 15 | 吉他 +4/hr, 鼓 +3/hr, 熟练度 +4%/hr | 《地下室之光》熟练度 ≥ 40% | 失恋后写的情感代表作 |
| 《404》 | 中等 (6) | 吉他、贝斯、鼓 | 吉他 ≥ 50, 贝斯 ≥ 20, 鼓 ≥ 30 | 各乐器 +5/hr, 熟练度 +3.5%/hr | 《锈》熟练度 ≥ 50% 且 鼓技能 ≥ 25 | 危机后全员完成的韧性之歌 |
| 《噪音祷告》 | 困难 (8) | 吉他、贝斯、鼓、主唱 | 吉他 ≥ 70, 贝斯 ≥ 40, 鼓 ≥ 50, 主唱 ≥ 40 | 各乐器 +6/hr, 熟练度 +3%/hr | 所有已解锁曲目平均熟练度 ≥ 50% | 第一首需要全员配合的高难度曲目 |
| 《末班地铁》 | 地狱 (10) | 吉他、贝斯、鼓、主唱 | 吉他 ≥ 90, 贝斯 ≥ 60, 鼓 ≥ 70, 主唱 ≥ 60 | 各乐器 +8/hr, 熟练度 +2%/hr | 《噪音祷告》熟练度 ≥ 70% 且 吉他技能 ≥ 80 | MVP 终极目标，地下之王的野心 |

### States and Transitions

每首曲目在其生命周期中存在三种离散状态：

| 状态 | 触发条件 | 玩家可见效果 |
|------|----------|-------------|
| **锁定 (Locked)** | 解锁条件未满足 | 曲目在列表中显示为灰色，名称被涂鸦覆盖，显示"???"和解锁提示 |
| **可排练 (Rehearsable)** | 已解锁，但熟练度 < 60% | 可加入排练计划，显示当前熟练度和等级标签 |
| **可演出 (Performance Ready)** | 熟练度 ≥ 60% | 排练效率加成生效，可被选为演出曲目，显示"可演出"徽章 |

**状态转换图**：

```
        解锁条件满足
   ┌──────────────────┐
   │                  │
   │     锁定         │─────────────► 可排练
   │                  │  熟练度≥60%
   └──────────────────┘        ┌─────────────► 可演出
                               │
                               │  熟练度<60%
                               └─────────────► 可排练
```

**状态降级规则**：曲目解锁后不会重新锁定。熟练度也不会下降（放置游戏，进度不可逆）。

### Interactions with Other Systems

**输入（本系统接收的数据）**：

| 来源系统 | 输入数据 | 作用 |
|----------|----------|------|
| 角色属性系统 | 乐器技能值 | 解锁条件判定：检查技能是否达到门槛 |
| 放置排练系统 | 排练时长、角色状态 | 熟练度计算输入（但计算公式由本系统定义） |
| 进度解锁系统 | 声誉值、进度里程碑 | 解锁条件判定（预留接口） |

**输出（本系统提供的数据）**：

| 目标系统 | 输出数据 | 作用 |
|----------|----------|------|
| 放置排练系统 | 曲目难度、所需乐器、排练收益参数、熟练度成长公式 | 计算排练成果 |
| 演出判定系统 | 曲目难度等级、熟练度百分比、熟练度等级标签 | 演出评分的原始输入 |
| 进度解锁系统 | 曲目解锁事件 | 触发解锁相关的进度节点 |
| UI 系统 | 曲目列表、解锁状态、熟练度显示 | 玩家界面展示 |

## Formulas

### 1. 默认技能门槛公式 (Default Skill Threshold)

The default skill threshold formula is defined as:

`default_threshold = difficulty_value × 10`

**Variables:**
| Variable | Symbol | Type | Range | Description |
|----------|--------|------|-------|-------------|
| difficulty_value | diff | int | 1–10 | 曲目内部难度数值 |
| default_threshold | threshold | int | 10–100 | 默认技能门槛 |

**Override Rule:** 当 `use_default_threshold = false` 时，`required_instruments` 中手动指定的值覆盖此公式。

**Example:** 难度值 6（中等）→ `default_threshold = 6 × 10 = 60`

### 2. 熟练度成长公式 (Proficiency Growth)

The proficiency growth formula is defined as:

`proficiency_gained = base_proficiency_rate × curve_multiplier × rehearsal_hours × mood_multiplier`

**Variables:**
| Variable | Symbol | Type | Range | Description |
|----------|--------|------|-------|-------------|
| base_proficiency_rate | base | float | 1.0–10.0 | 歌曲定义的基础熟练度成长率（%/小时） |
| current_proficiency | current | int | 0–100 | 当前熟练度百分比 |
| rehearsal_hours | hours | float | ≥0 | 排练小时数 |
| mood_multiplier | mood | float | 0.5–1.2 | 心情修正（来自角色属性系统：低落×0.5, 正常×1.0, 亢奋×1.2） |

**Curve Multiplier 计算（按曲线类型）：**

| 曲线类型 | 公式 | 说明 |
|----------|------|------|
| 线性 (linear) | `curve_mult = 1.0` | 恒定成长 |
| 递减 (diminishing) | `curve_mult = 1.0 - (current / 100) × 0.5` | 熟练度越高成长越慢，0%时×1.0，100%时×0.5 |
| 阶梯 (stepped) | `curve_mult = 1.0 + level_bonus` | 每跨越一个等级阈值，短暂加成 |

**阶梯等级加成：**
| 跨越的阈值 | bonus |
|-----------|-------|
| 生疏 → 熟悉 | +0.3 |
| 熟悉 → 熟练 | +0.2 |
| 熟练 → 精通 | +0.15 |
| 精通 → 大师 | +0.1 |

**Output Range:** 0% 到 100%（硬封顶）
**Example:** 《锈》base=4%/hr，线性曲线，心情正常，排练 2 小时 → `proficiency_gained = 4 × 1.0 × 2 × 1.0 = 8%`

### 3. 排练效率修正公式 (Rehearsal Efficiency Modifier)

The rehearsal efficiency modifier from song proficiency is defined as:

`proficiency_efficiency_modifier = 1.0 + proficiency_bonus`

**Proficiency Bonus 计算：**
| 熟练度等级 | bonus |
|-----------|-------|
| 生疏 (0–20%) | -0.20 |
| 熟悉 (21–40%) | 0.00 |
| 熟练 (41–60%) | +0.10 |
| 精通 (61–80%) | +0.20 |
| 大师 (81–100%) | +0.30 |

**Usage:** 此修正值与角色属性系统的 `rehearsal_efficiency` 公式相乘，共同决定最终排练效率。

**Example:** 林野当前排练效率为 1.3（技术型加成），曲目熟练度为精通（+0.20）→ `final_efficiency = 1.3 × 1.2 = 1.56`

### 4. 演出评分加成公式 (Performance Score Bonus)

The performance score bonus from song proficiency is defined as:

`performance_bonus = proficiency_level_bonus`

**Proficiency Level Bonus 计算：**
| 熟练度等级 | bonus |
|-----------|-------|
| 生疏 | 0% |
| 熟悉 | +5% |
| 熟练 | +10% |
| 精通 | +15% |
| 大师 | +20% |

**Note:** 此加成由演出判定系统读取并应用。Song Database 只提供加成值，不定义评分公式。

### 5. 解锁判定公式 (Unlock Evaluation)

The unlock evaluation formula is defined as:

`is_unlocked = evaluate_all(conditions, logic)`

**Where:**

`evaluate_all(conditions, AND)` returns `true` if **all** conditions evaluate to `true`.

`evaluate_condition(condition)` returns:
- `SKILL_THRESHOLD`: `character_skill[instrument] ≥ min_skill`
- `SONG_PROFICIENCY`: `song_proficiency[song_id] ≥ min_proficiency`
- `REPUTATION_MILESTONE`: `reputation ≥ min_reputation`

**Example:** 《404》的解锁条件：
- 条件 A: `song_proficiency["rust"] ≥ 50` → true
- 条件 B: `character_skill["drums"] ≥ 25` → true
- `is_unlocked = true AND true = true`

## Edge Cases

- **If 角色技能恰好等于门槛值（如吉他=50，门槛=50）**：判定为**满足**，解锁生效。门槛是"大于等于"，不是"大于"。
- **If 多首曲目同时满足解锁条件**：按曲目 ID 字母顺序依次解锁，每首触发独立的解锁提示，避免信息过载。
- **If 解锁条件中的前置曲目已被移除或不存在**：条件自动判定为 `false`，曲目保持锁定。日志记录警告。
- **If 熟练度计算结果超过 100%**：硬封顶至 100%，溢出部分丢弃，不累积到下一等级。
- **If 熟练度计算结果为小数**：向下取整到整数百分比（如 47.8% → 47%）。
- **If 角色处于亢奋状态排练**：心情加成（×1.2）直接应用于熟练度成长公式，最高可获得 `base × 1.2` 的成长速度。
- **If 所有已解锁曲目的熟练度都达到 100%**：曲目数据库本身不做任何特殊处理。进度解锁系统可据此触发"曲目全精通"成就（如有）。
- **If 一首需要 4 种乐器的曲目，但乐队当前只有 2 个角色**：曲目仍可解锁（解锁条件与乐队人数无关），但排练时缺少的乐器部分不产出对应技能成长。放置排练系统负责处理此情况。
- **If 一首歌的 `use_default_threshold = false` 但手动值未指定**：回退到默认映射（`difficulty_value × 10`），并在编辑器中显示警告。
- **If 解锁条件包含未来系统（如声誉）但该系统尚未实现**：条件默认判定为 `false`，并在日志中标记为"依赖未就绪系统"。

## Dependencies

**上游依赖**：
| 系统 | 依赖数据 | 必要性 |
|------|----------|--------|
| 角色属性系统 | 乐器技能值 | 硬依赖 — 解锁判定需要 |
| 放置排练系统 | 排练时长、角色状态 | 软依赖 — 本系统只定义公式，状态由对方提供 |
| 进度解锁系统 | 声誉值、进度里程碑 | 软依赖 — 预留接口，MVP 可能暂不启用 |

**下游被依赖**：
| 系统 | 依赖数据 | 数据流向 |
|------|----------|----------|
| 放置排练系统 | 曲目参数、成长公式 | Song DB → 排练计算 |
| 演出判定系统 | 难度等级、熟练度加成 | Song DB → 演出评分 |
| 进度解锁系统 | 解锁事件 | Song DB → 进度节点 |
| UI 系统 | 曲目列表、状态、熟练度 | Song DB → 界面展示 |

## Tuning Knobs

| 参数名 | 当前值 | 安全范围 | 过低效果 | 过高效果 | 关联系统 |
|--------|--------|----------|----------|----------|----------|
| `difficulty_threshold_multiplier` | 10 | 8–15 | 门槛过低，玩家轻松解锁高难度曲目 | 门槛过高，玩家长期卡关 | 进度解锁 |
| `base_proficiency_range` | 2–5 %/hr | 1–8 | 熟练度成长过慢，玩家失去成就感 | 成长过快，放置感消失 | 放置排练 |
| `proficiency_cap` | 100 | 100 | 不适用 | 超出后无上限，破坏平衡 | 放置排练 |
| `performance_ready_threshold` | 60 | 40–80 | 太容易演出，缺乏"准备感" | 太难演出，玩家挫败 | 演出判定 |
| `linear_curve_efficiency` | 1.0 | 0.8–1.2 | 线性曲线失去竞争力 | 线性过快，递减/阶梯无意义 | 放置排练 |
| `diminishing_curve_max_penalty` | 0.5 | 0.3–0.7 | 递减惩罚不足，后期仍成长快 | 递减惩罚过重，后期几乎不涨 | 放置排练 |
| `step_level_bonus_max` | 0.3 | 0.1–0.5 | 阶梯突破感不足 | 阶梯奖励过强，玩家刻意卡点 | 放置排练 |
| `proficiency_efficiency_bonus_max` | +0.30 | +0.10–+0.50 | 高熟练度优势不明显 | 低熟练度曲目无人问津 | 放置排练 |
| `performance_bonus_max` | +20% | +10–+30% | 熟练度对演出影响微弱 | 演出评分过度依赖熟练度 | 演出判定 |

## Visual/Audio Requirements

- **曲目卡片**：每首歌的卡片像一张从旧传单上剪下来的纸片，边缘参差不齐。已解锁的卡片颜色鲜明（脏黄/铁锈红），锁定的卡片被灰色涂鸦覆盖。
- **难度标签**：用视觉符号表示——入门=单根琴弦，简单=两根交叉，中等=三根编织，困难=断裂后重新接上的琴弦，地狱=燃烧的琴弦。
- **熟练度进度条**：手撕纸条风格的进度条，填充部分像墨水晕染。颜色随等级变化：生疏=褪色灰，熟悉=淡黄，熟练=烟黄，精通=铁锈红，大师=荧光粉（唯一使用荧光色的数据元素，代表" mastered"）。
- **解锁动画**：锁定状态解除时，涂鸦像被雨水冲掉一样褪去，露出下面的曲目信息。伴随一声吉他弦的轻拨。
- **状态徽章**："可演出"徽章像一枚生锈的图钉，钉在曲目卡片的右上角。

## UI Requirements

- **曲目列表界面**：纵向滚动的卡片列表，每张卡片显示曲目名称、难度标签、熟练度进度条、当前等级标签。锁定曲目显示在最后，以灰色涂鸦覆盖。
- **曲目详情面板**：点击卡片后展开的详情页，显示完整背景故事、所需乐器及当前乐队技能对比（达标/未达标可视化）、排练收益预览、解锁条件进度。
- **排练计划选择器**：在设置排练计划时，曲目列表需要标记"当前乐队可排练"（所有所需乐器有角色覆盖）和"有角色技能未达标"（可选但效率降低）。
- **筛选与排序**：支持按难度、熟练度、解锁状态筛选；支持按解锁顺序/难度/熟练度排序。
- **解锁提示**：新曲目解锁时，主界面显示手写风格的浮动提示："解锁新曲目：《XXX》——去排练室看看！"

## Acceptance Criteria

- **GIVEN** 一首难度值为 6 的歌曲且 `use_default_threshold = true`，**WHEN** 计算默认技能门槛，**THEN** 门槛为 60
- **GIVEN** 一首 base_proficiency_rate = 4%/hr、线性曲线、心情正常的歌曲，**WHEN** 排练 2 小时，**THEN** 熟练度增加 8%
- **GIVEN** 当前熟练度为 75%（精通等级）的歌曲，**WHEN** 计算排练效率修正，**THEN** 修正值为 +20%
- **GIVEN** 《锈》熟练度为 60% 且鼓技能为 30，**WHEN** 判定《404》解锁条件，**THEN** 解锁成功（两个条件均满足）
- **GIVEN** 《锈》熟练度为 30% 且鼓技能为 30，**WHEN** 判定《404》解锁条件，**THEN** 解锁失败（熟练度条件未满足）
- **GIVEN** 一首熟练度为 98% 的歌曲，**WHEN** 排练后计算新熟练度为 102%，**THEN** 新熟练度硬封顶至 100%
- **GIVEN** 一首需要吉他+贝斯的歌曲，**WHEN** 乐队只有吉他手（无贝斯手），**THEN** 吉他部分正常产出技能成长，贝斯部分产出为 0
- **GIVEN** 所有已解锁曲目的平均熟练度 ≥ 50%，**WHEN** 判定《噪音祷告》解锁条件，**THEN** 解锁成功
- **GIVEN** 一首熟练度从 39%（熟悉）提升到 41%（熟练）的歌曲，**WHEN** 使用阶梯曲线，**THEN** 该次成长获得 +0.2 的阶梯加成
- **GIVEN** 一首锁定状态的歌曲，**WHEN** 其解锁条件满足，**THEN** 自动转换为可排练状态并触发解锁提示

## Open Questions

- **Q1**: 是否需要为每首歌定义"风格兼容性"（如摇滚曲目配摇滚乐队有加成）？这会增加策略深度但提高复杂度。
- **Q2**: 曲目熟练度达到大师后，是否需要"维持机制"（不排练会衰减），还是永久保持？放置游戏通常不衰减，但可能失去"维护感"。
- **Q3**: 是否需要"原创曲目"系统（玩家到一定进度后可以自定义歌名和参数），还是全部使用预设曲目？
- **Q4**: 高熟练度曲目的演出加成（+20%）是否过高？需要通过 playtest 验证。
- **Q5**: 解锁条件中的 OR 逻辑在 MVP 中是否需要实现，还是可以先只用 AND 简化？
